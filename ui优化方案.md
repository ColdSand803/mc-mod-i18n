# mc-mod-i18n 工作区 UI 优化方案（2026-05）

## 1. 目标

本方案只聚焦 **工作区主界面**，不讨论整站视觉重做，也不讨论引入前端框架。

目标是把当前工作区优化成一个更像“本地翻译工作台”的界面，让用户在一次任务中更顺地完成：

```text
选择输入 -> 调整配置 -> 运行预检 -> 开始翻译 -> 处理中观察状态 -> 结果审校 -> 导出产物
```

核心要求：

- 首屏看得懂
- 高级项不打脸
- 错误能快速定位
- 结果更适合人工复核
- 保持当前 `web.py` 单文件架构可持续演进

---

## 2. 当前工作区现状

基于当前仓库代码，工作区已经具备这些能力：

### 2.1 已有输入与配置能力

- 三种输入模式：
  - JAR
  - FTB Quests
  - JSON
- 可搜索的源语言 / 目标语言选择
- provider 选择
- pack format 选择
- 术语表文件上传
- 高级 API 设置折叠区
- 输出策略折叠区

### 2.2 已有流程支持

- 预检面板
- 进行中进度卡片
- 取消任务
- 结果页多视图
- 失败项重试
- 历史任务页
- 设置页：缓存、预设、翻译记忆、UI locale、术语表

### 2.3 当前不是“没 UI”，而是还存在这些问题

工作区的主要问题已经从“功能缺失”转向：

- 配置层级还能更清楚
- 主流程还不够线性
- 成本预估和风险提示还能更靠前
- 结果页虽然功能全，但对首次用户还是偏重

---

## 3. 本次方案的范围

本方案只管：

- 工作区首屏
- 预检
- 进行中状态
- 结果区入口与审校效率

本方案暂不展开：

- 设置页大改版
- 任务历史重构
- 全局主题体系重做
- 图标系统替换
- 拆分 `web.py` 为多文件

这些都可以后续再立项。

---

## 4. 工作区的主要问题

### 4.1 首屏信息密度仍偏高

虽然高级 API 设置已经折叠，但工作区首屏仍然把这些块连续堆在一起：

- 输入类型
- 文件选择
- 语言设置
- provider / pack format
- 术语表
- 预检
- 高级 API
- 输出策略

问题不是“太多”，而是缺少更强的阶段感。用户仍像是在填写一长张配置表，而不是走任务流程。

### 4.2 provider 与高级设置的联动还不够彻底

当前高级 API 区已经折叠，但用户切换 `copy` / `glossary` 时，工作区仍然保留“这是同一套复杂翻译设置”的心理模型。

更理想的状态是：

- 离线 provider 看起来明显更轻
- AI provider 看起来明显更重
- 不同 provider 的风险和成本提示能前置

### 4.3 预检在流程上还是“附属工具”

当前有预检面板，这是很好的基础；但从操作感上，它更像一个额外按钮，而不是翻译前的自然一步。

更理想的是：

- 用户在点击“开始生成”前，先自然看到预检摘要
- 用户知道待翻译条目数、风险数、缓存命中和预计成本

### 4.4 工作区的“当前任务”反馈仍偏散

现在顶部 header、状态框、loading card 都能反映任务状态，但信息分布有点散：

- 顶部有任务 ID
- 状态框有阶段文本
- loading card 有详细进度

信息量够了，但主次还可以更收束。

### 4.5 结果区入口已经很全，但首次决策路径仍偏长

当前结果页有：

- 输出 / 质量 / 性能指标
- 工作区 / 报告 / API 日志 / 硬编码等入口
- 表格 / 对比视图
- 失败项重试

问题不是缺功能，而是完成任务后用户的第一决策应该更明确：

1. 能不能下载
2. 有没有失败项
3. 哪些条目需要人工复核

---

## 5. 优化原则

1. **把工作区改成流程界面，不是参数墙**
2. **把“预检 -> 开始生成”变成默认路径**
3. **让不同 provider 的复杂度被看见**
4. **让失败和风险比普通成功项更靠前**
5. **尽量复用现有组件，不搞大拆**

---

## 6. 建议的工作区结构

建议把工作区重组成 5 个连续区块：

### 6.1 Step 1: 输入来源

内容：

- 输入类型切换
- 对应文件上传控件
- 当前已选择文件摘要

目标：

- 用户先确认“我要处理什么”

### 6.2 Step 2: 语言与翻译方案

内容：

- 源语言
- 目标语言
- provider
- pack format
- provider 风险提示

目标：

- 用户确认“我要翻成什么、用什么方式翻”

### 6.3 Step 3: 可选增强项

内容：

- 术语表
- 输出策略
- 高级 API 设置

目标：

- 把“不是必须填”的东西放到第二层

### 6.4 Step 4: 预检与启动

内容：

- 预检摘要
- 风险与 warning
- 开始生成 / 返回修改

目标：

- 真正把预检变成启动前一环

### 6.5 Step 5: 当前任务与结果

内容：

- 进行中进度卡片
- 完成后结果入口

目标：

- 让用户在同一个主视角里观察整个任务生命周期

---

## 7. P0：首轮必须做

### 7.1 给工作区增加“步骤标题”

不是要做复杂 wizard，而是给现有表单卡片加明确步骤语义。

建议文案：

- 步骤 1：选择输入
- 步骤 2：选择语言与翻译器
- 步骤 3：调整高级选项
- 步骤 4：运行预检并开始

价值：

- 用户读起来更像流程
- 比现在的多个平级卡片更容易理解顺序

### 7.2 provider 风险提示前置

当用户切换不同 provider 时，工作区即时显示提示条：

- `copy`
  - 不翻译，仅复制原文，适合验证流程
- `glossary`
  - 离线规则翻译，稳定但覆盖有限
- `openai-compatible`
  - 质量高，可配置，需 API Key，可能有费用
- `anthropic-compatible`
  - 同上
- 未来 `deep-free`
  - 免费试用，不稳定，失败会回退

要求：

- 提示条要出现在 provider 区下方
- 不要藏到折叠面板里

### 7.3 预检摘要默认可见

当前预检已经是独立卡片，建议进一步做成：

- 未运行时：
  - 显示“建议先运行预检”
- 已运行后：
  - 固定显示摘要 chips
  - 显示 warning / blocker 数
  - 如果无阻断，突出“可开始生成”

最低要求：

- 开始生成前，用户能一眼看到：
  - 输入文件数
  - 待翻译条目数
  - 预计输出数
  - warning / blocker 数

### 7.4 开始生成按钮做成带上下文的主操作

当前按钮文案统一是“开始生成”，可以再增强：

- 默认：
  - `开始生成`
- 预检有阻断时：
  - `存在阻断，需先处理`
- 预检已完成且无阻断：
  - `开始生成（已通过预检）`

这样用户不需要回看一堆状态文本。

### 7.5 当前任务卡片与状态框收束

建议明确分工：

- 顶部 header
  - 只显示当前任务 ID / 全局状态

- 工作区状态框
  - 只显示一句摘要状态

- loading card
  - 负责详细信息：
    - 当前阶段
    - 当前文件
    - 文件进度
    - 请求进度
    - 缓存命中
    - 重试原因

避免三个地方都讲同样的话。

---

## 8. P1：第二轮优化结果审校

### 8.1 结果区增加“优先处理区”

当前结果指标已经分成输出 / 质量 / 性能，这很好。

建议再增加一个紧贴摘要的优先区：

- 如果有失败项：
  - 显示失败项数量
  - 提供“只看失败项”入口
  - 提供“重试失败项”入口

- 如果没有失败项但有风险项：
  - 提供“只看风险项”入口

目标：

- 结果页第一屏先告诉用户“接下来该处理什么”

### 8.2 把对比视图提升为审校主视图

现在结果页已经有表格 / 对比双视图，这是很好的基础。

建议：

- 当存在风险项时，默认落在对比视图
- 当都是常规成功项时，默认保留表格视图

原因：

- 对比视图更适合人工复核
- 表格视图更适合扫量

### 8.3 失败项入口统一

当前失败处理能力分散在：

- 条件筛选
- 报告视图
- 重试按钮

建议统一成一个更显眼的入口：

- “失败 / 风险项工作区”

第一阶段不一定要新增完整新面板，但至少要在结果页头部统一收口入口。

---

## 9. P1：工作区与设置页的边界优化

现在有些内容已经挪到设置页，这是对的，比如：

- 缓存目录
- 配置预设
- 翻译记忆
- UI locale
- 术语表管理

工作区后续应继续保持“任务内必要配置优先”的边界：

### 工作区保留

- 输入
- 语言
- provider
- 本次任务相关高级参数
- 本次任务术语表上传
- 输出策略

### 设置页保留

- 预设管理
- 全局术语表维护
- 翻译记忆维护
- 缓存维护
- UI 语言包维护

不要再把更多“长期配置管理功能”塞回工作区。

---

## 10. P2：后续可做但不急

### 10.1 工作区 compact / expert 模式

可以给工作区增加两种显示模式：

- `标准模式`
  - 只显示常用配置
- `专家模式`
  - 展开全部技术参数

这比持续往首屏加说明更干净。

### 10.2 provider 成本估计

对 AI provider，可在预检后显示：

- 待翻译条目数
- 预计 batch 数
- 推荐并发

不一定给美元成本，但至少给规模感。

### 10.3 工作区恢复上次任务配置

这个功能可以增强效率，但要谨慎：

- 只恢复非敏感字段
- 不恢复 API Key 明文

---

## 11. 不建议现在做的事

- 不建议把工作区改成真正多页 wizard
  - 当前本地工具需要高效率，不适合每一步都切屏

- 不建议引入 React/Vue
  - 当前问题是信息结构，不是前端框架能力

- 不建议把结果页继续塞更多 metric 卡片
  - 现在更需要的是入口优先级，不是更多数字

- 不建议把所有 provider 的配置做成完全不同的动态表单
  - 维护成本太高
  - 用提示和折叠层级就够了

---

## 12. 最小可落地版本

如果只做一轮工作区优化，我建议先做这 5 项：

1. 工作区卡片加步骤语义，形成线性流程
2. provider 区增加即时风险 / 成本提示
3. 预检摘要默认可见，强化“先预检再生成”
4. 开始生成按钮文案跟预检状态联动
5. 结果页增加“失败 / 风险优先处理区”

这 5 项不会推翻现有结构，但能明显提升可理解性和可操作性。

### 第一阶段：工作区流程清晰化

- [ ] 给工作区 4 个核心卡片加步骤标题
- [ ] 在 provider 下方增加风险提示条
- [ ] 强化预检摘要的默认展示

- [ ] 明确 header / status / loading card 的分工
- [ ] 优化 loading card 的详细信息排列

- [ ] 新增失败 / 风险优先处理区
- [ ] 让有风险任务默认落到对比视图

---

## 14. 结论

当前工作区已经不是“缺功能”的阶段，而是“需要更强流程感”的阶段。

最值得做的不是大改视觉，而是：

- 让用户更快知道先做什么
- 让不同 provider 的复杂度更透明
- 让预检真正成为启动前步骤
- 让结果页先服务于“下载 / 排错 / 复核”这三个核心决策

这会比继续堆功能或重写框架更有效。
