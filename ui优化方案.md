# mc-mod-i18n UI 优化方案

> 目标：在不引入大型前端框架、不重写产品结构的前提下，把当前本地 Web UI 从“功能都摆出来”优化为“流程清楚、错误可定位、结果可审校、离线可交付”的桌面化工具界面。

## 1. UI 定位

当前项目不是营销站，也不是轻量表单页，而是面向 Mod 汉化、批量任务、API 调试和结果审校的本地工作台。UI 应该保持高信息密度，但不能把所有配置和结果平铺到用户面前。

推荐风格：

- **产品类型**：本地桌面工具 / 翻译工作台 / 任务型 dashboard。
- **视觉关键词**：清晰、克制、专业、可扫描、高密度。
- **不建议方向**：大 hero、营销式卡片堆叠、过强渐变、过多装饰动效。
- **技术前提**：继续使用当前 `web.py` 内联 HTML/CSS/JS 架构，先优化交互和信息结构，再考虑拆文件。

## 2. 当前 UI 已有基础

当前 UI 已经具备这些基础，后续应复用而不是推倒：

- 左侧 sidebar + 主内容区的桌面化布局。
- 深色/浅色/跟随系统主题。
- CSS 变量体系，已有状态色、表格色、卡片色、动效变量。
- 文件上传、翻译器配置、任务进度、中断任务。
- 结果指标、语言结果表、报告视图、硬编码视图、API 日志视图。
- 搜索、防抖、分页、错误项重试、硬编码映射导入/导出。
- `prefers-reduced-motion` 已有基础适配。

核心问题不是“缺少 UI”，而是信息层级和任务流还可以更顺。

## 3. 当前主要问题

### 3.1 配置区一次性暴露过多

工作区里同时展示基础语言、provider、pack_format、API URL、API Key、并发、重试、batch、timeout、debug log、覆盖策略、硬编码扫描。新用户无法快速判断哪些是必须填，哪些是高级项。

### 3.2 导航重复

左侧 sidebar 和顶部 tabs 都有“工作区 / 报告 / 硬编码 / API 日志”。这会占用顶部空间，也会让用户不确定哪个导航是主导航。

### 3.3 顶部搜索语义不明确

`top-search` 显示“搜索工作区...”，但它不是清晰的全局搜索入口。当前真正有效的搜索分散在语言结果、报告、硬编码和 API 日志内部。

### 3.4 缺少翻译前预览

用户点击“开始生成”前，不知道会处理多少 JAR、多少条目、多少已有译文会跳过、预计发起多少 API batch、是否命中缓存、是否存在预检风险。

### 3.5 结果指标并列过多

完成后同时展示 JAR、语言文件、新增翻译、报告条目、耗时、候选、硬编码、异常缺失。所有指标权重接近，用户需要自己判断哪个重要。

### 3.6 表格编辑体验偏拥挤

语言结果表直接放 textarea，适合少量编辑；一旦长文本、多列、多条目同时出现，表格会变得拥挤。用户很难专注审校某一条译文和对应错误。

### 3.7 API 日志偏原始调试数据

API 日志现在能看输入输出，但更像开发者原始 JSON 面板。用户排错时更需要先看到错误时间线：哪次请求失败、HTTP 状态、重试等待、最终失败原因。

### 3.8 自定义控件可访问性还可加强

自定义 select、图标按钮、弹窗和 tab 需要补齐键盘交互、ARIA 语义和明确的 focus 状态。当前已有部分 `aria-current`、`aria-modal`，但还不完整。

### 3.9 外部图标和字体影响离线体验

当前 UI 引用 Remix Icon CDN 和 Google Fonts。打包成 Windows exe 或断网使用时，图标/字体可能不可用，影响本地工具的可靠性。

## 4. 优化原则

1. **渐进披露**：先展示必须项，高级参数折叠。
2. **任务优先**：界面围绕“扫描 -> 确认 -> 翻译 -> 审校 -> 导出”组织。
3. **错误优先**：失败项、缺失占位符、API 错误要比普通成功项更容易到达。
4. **结果可审校**：表格负责扫视，详情面板负责编辑和修复。
5. **本地可靠**：断网、exe、窄屏、长文本都要正常。
6. **少改架构**：先在 `src/mc_mod_i18n/web.py` 内完成 UI 行为优化；拆分文件放到后续阶段。

## 5. P0：第一轮必须做

### 5.1 重整工作区配置

**目标**

让用户第一次打开页面时只看到完成翻译所需的最小路径。

**调整方案**

- 保留基础区：
  - JAR 上传
  - 源语言
  - 目标语言
  - 翻译器
  - pack_format
  - 开始扫描 / 开始生成
- 折叠高级 API 区：
  - API URL
  - API Key
  - API Key 环境变量
  - 并发数
  - 重试次数
  - batch size
  - timeout
  - debug log
- 折叠输出策略区：
  - 覆盖已有译文
  - 跳过已汉化 JAR
  - 扫描硬编码英文
  - 自定义术语表

**落地位置**

- `src/mc_mod_i18n/web.py` 的 `INDEX_HTML` 表单区域。
- 复用 `.form-card`，新增 `.section-toggle` / `.advanced-panel`。

**验收标准**

- 默认首屏能看清上传、语言、翻译器、开始按钮。
- 切换 `copy` / `glossary` 时不展示无关 API 参数。
- 切换 AI provider 时才展开或提示 API 设置。

### 5.2 去掉重复顶部 tabs

**目标**

建立单一主导航，释放顶部空间。

**调整方案**

- 左侧 sidebar 保留主导航：工作区、翻译报告、硬编码、API 日志。
- 顶部 header 改为：
  - 当前任务状态
  - 当前任务耗时
  - 主题切换
  - 后续可放任务历史入口
- 移除或隐藏顶部重复 tab。
- `top-search` 暂时移除；后续如做全局搜索，再重新设计。

**验收标准**

- 页面只有一个主导航系统。
- 当前视图标题和任务状态在顶部清楚可见。

### 5.3 增加扫描预览

**目标**

让用户在真正调用 API 前知道成本和风险。

**调整方案**

点击按钮后先进入“扫描预览”状态：

- JAR 数量
- 源语言文件数量
- 待翻译条目数
- 已有目标译文数量
- 预计 API batch 数
- 缓存命中数量
- 预检 warning / error 数量
- 是否包含 nested JAR
- 是否启用硬编码扫描

预览卡片提供两个操作：

- 开始翻译
- 返回修改配置

**落地位置**

- 前端新增预览卡片区域。
- 后端需要配合新增 preview/dry-run API；如果后端暂时不做，可先在 UI 方案里预留入口，按钮仍直接翻译。

**验收标准**

- 用户不会在完全未知成本的情况下直接发起 AI 翻译。
- 预检错误项在开始前可见。

### 5.4 重构任务进度卡片

**目标**

让进行中的任务状态可读，不只是显示一段文本。

**调整方案**

把 loading card 拆成四段：

- 阶段：上传 / 扫描 / 处理 JAR / 翻译 / 重试 / 写入产物 / 完成
- 当前文件：显示正在处理的 JAR
- 进度：JAR 进度、API batch 进度、缓存命中
- 异常：当前重试原因、等待秒数、失败数量

**验收标准**

- 多 JAR 任务中用户能看出当前处理到哪个 JAR。
- API 429 或超时时，状态明确显示正在等待重试，而不是像卡住。

### 5.5 重排完成结果指标

**目标**

把结果从“8 个并列数字”改成按决策价值分组。

**调整方案**

分为三组：

- **输出**
  - 资源包
  - HTML 报告
  - hardcoded map
- **质量**
  - 成功翻译
  - 已有译文
  - 校验失败
  - API 失败
  - incomplete
- **性能**
  - 耗时
  - 缓存命中
  - 实际翻译 JAR
  - API 重试次数

按钮顺序建议：

1. 下载资源包
2. 查看失败项
3. 打开报告
4. 查看 API 日志
5. 硬编码映射

**验收标准**

- 用户完成任务后第一眼能找到下载资源包。
- 如果有失败项，失败入口比普通报告更醒目。

## 6. P1：第二轮提升审校效率

### 6.1 语言结果改为“表格 + 详情面板”

**目标**

表格负责快速扫描，详情面板负责审校和编辑。

**调整方案**

- 表格列减少为：
  - 状态
  - JAR
  - Mod ID
  - Key
  - 原文摘要
  - 译文摘要
  - 操作
- 点击行后打开右侧详情面板：
  - 完整原文
  - 完整译文 textarea
  - 校验错误
  - 占位符对比
  - 回退原文
  - 保存修改
  - 单条重试

**验收标准**

- 长文本不会撑破表格。
- 用户可以专注修改单条译文。

### 6.2 失败项优先队列

**目标**

把失败项从普通报告中提出来，形成可处理队列。

**调整方案**

- 新增“失败项”筛选入口。
- 按失败类型分组：
  - `api_failed`
  - `failed`
  - `incomplete`
  - `jar_failed`
- 每组显示数量和建议动作。
- 支持只重试 API 失败项。
- 支持导出失败项 JSON。

**验收标准**

- 有失败时，用户不需要在所有报告条目中搜索。
- 重试完成后，失败数量和资源包内容同步更新。

### 6.3 API 日志改为时间线

**目标**

先给人看懂，再给原始 JSON。

**调整方案**

API 日志默认显示时间线：

- 时间
- provider
- model
- batch size
- 状态码
- 重试次数
- 耗时
- 错误摘要

点击一行再展开：

- request headers
- request body
- response headers
- response preview
- raw JSON

**验收标准**

- 用户能快速判断是 API Key、限流、超时、模型返回格式还是缺失条目。

### 6.4 硬编码工作台增强

**目标**

降低硬编码候选的人工筛选成本。

**调整方案**

- 分类 tabs 固定在表格顶部。
- 增加风险优先排序：high -> medium -> low。
- 增加“只看已填写 / 未填写 / 校验异常”。
- 选中项数量固定显示在工具栏。
- AI 翻译所选后显示成功/失败数量。
- 导出前弹出摘要：导出多少条、跳过多少空译文。

**验收标准**

- 用户能快速处理 high risk 的候选。
- 导出的 `hardcoded-map.json` 不包含空译文。

## 7. P2：第三轮补完整体验

### 7.1 任务历史入口

**目标**

让用户能回看已完成任务，而不是只依赖当前内存状态。

**调整方案**

- 顶部 header 增加“历史任务”入口。
- 历史列表展示：
  - 时间
  - 文件数量
  - provider
  - 状态
  - 耗时
  - 失败数量
  - 下载/打开报告

**前提**

需要后端先做任务状态持久化，否则 UI 只能显示当前会话。

### 7.2 本地化图标和字体

**目标**

保证 exe 和断网环境下 UI 完整。

**调整方案**

- Remix Icon 改为本地 CSS/font 文件，或替换为内联 SVG 图标集合。
- Google Fonts 改为本地字体，或优先使用系统字体栈。
- PyInstaller spec 打包 UI 资源。

**验收标准**

- 断网启动 UI，图标不丢失，字体回退稳定。

### 7.3 自定义控件可访问性

**目标**

键盘和屏幕阅读器可用。

**调整方案**

- 自定义 select 支持：
  - Enter / Space 打开
  - ArrowUp / ArrowDown 选择
  - Esc 关闭
  - aria-expanded
  - aria-controls
  - role=listbox / option
- 图标按钮补 `aria-label`。
- 弹窗关闭后焦点回到触发按钮。
- 所有动态状态更新放入 `aria-live` 区域。

**验收标准**

- 不用鼠标也能完成一次基础翻译。
- 焦点始终可见，不会丢失。

### 7.4 CSS token 收敛

**目标**

减少后续维护成本。

**调整方案**

- 保留核心语义变量：
  - background
  - surface
  - panel
  - text
  - muted
  - line
  - accent
  - success
  - warning
  - danger
- 卡片阴影、表格、字段、按钮状态统一从这些变量派生。
- 避免新增一次性颜色。

**验收标准**

- 深浅色切换不需要为每个组件单独补颜色。
- 新组件能直接使用现有 token。

## 8. 建议实施顺序

### 第一阶段：工作流清晰化

- [ ] 折叠高级 API 设置和输出策略。
- [ ] 移除顶部重复 tabs。
- [ ] 移除或重定义顶部搜索。
- [ ] 重构进行中任务卡片。
- [ ] 重排完成结果指标和操作按钮。

**验收**

- 首屏只呈现最小翻译路径。
- 完成任务后能立即找到下载、失败项和报告。

### 第二阶段：翻译前预览

- [ ] 设计扫描预览卡片。
- [ ] 增加后端 preview/dry-run 数据。
- [ ] 显示待翻译条目、缓存命中、预计 batch、风险数量。
- [ ] 支持确认后再开始翻译。

**验收**

- 用户能在 API 调用前明确成本和风险。

### 第三阶段：结果审校

- [ ] 语言结果改为表格摘要。
- [ ] 增加右侧详情面板。
- [ ] 增加失败项队列。
- [ ] API 日志改为时间线 + 原始详情。
- [ ] 硬编码工作台补风险排序和导出摘要。

**验收**

- 长文本审校不挤表格。
- 失败项可以集中处理。

### 第四阶段：交付稳定性

- [ ] 本地化图标和字体。
- [ ] 补键盘交互和 ARIA。
- [ ] 验证 375px、768px、1024px、1440px。
- [ ] 验证浅色、深色、跟随系统。
- [ ] 验证断网 exe 启动。

**验收**

- UI 在桌面、窄屏、离线、exe 环境下都可用。

## 9. 不建议现在做的事

- 不建议引入 React/Vue/Next。当前瓶颈不是框架，而是信息结构和任务流。
- 不建议做营销式首页。用户打开工具后应该直接进入工作台。
- 不建议继续堆更多 metric 卡片。指标应分组，不应平铺扩张。
- 不建议在表格里塞更多输入控件。长文本编辑应进入详情面板。
- 不建议依赖外部 CDN。这个工具的交付目标包含 Windows exe 和本地离线场景。

## 10. 最小可落地版本

如果只做一轮 UI 优化，建议做这 6 项：

1. 基础设置 / 高级设置 / 输出策略分组折叠。
2. 移除顶部重复 tabs，顶部只保留任务状态和主题切换。
3. 完成结果按“输出 / 质量 / 性能”分组。
4. 失败项入口前置，支持只看失败项。
5. API 日志增加错误时间线。
6. 本地化图标和字体，保证 exe 离线可用。

这 6 项完成后，用户路径会明显清楚，排错效率会提升，打包后的可用性也更可靠。
