# mc-mod-i18n 拓展规划

## 目标

把当前工具从“能生成翻译结果”推进到“可稳定交付、可排查、可复用”的本地翻译工作台。后续拓展优先解决真实使用中的断点：页面交互是否可用、API 配置是否正确、翻译前是否能预判风险、失败项是否能低成本重试、历史结果是否能找回。

## 规划原则

- **先稳定，再扩展**：优先补自检、连接测试、错误定位能力，再做大型功能。
- **围绕本地工具定位**：保持零账号、低部署成本，不引入云端多用户系统。
- **每个功能都要可验证**：新增功能同时补单元测试、HTML 契约测试或浏览器级自检。
- **减少重复劳动**：历史任务、失败重试、术语表管理都服务于可复用翻译资产。
- **保留 CLI/Web 一致性**：Web 新能力尽量能映射到 CLI 参数或底层公共函数。

## 当前基线

已具备：

- Mod JAR 语言文件翻译。
- FTB Quests 输入处理。
- 直接 JSON 语言文件处理。
- Web UI 语言切换和扩展语言包导入导出。
- 资源包、报告、API 调试日志下载。
- 硬编码扫描和硬编码映射工作台。
- 共享缓存、配置哈希和基础断点能力。

近期已补强：

- 修复内联 JS 语法错误导致的下拉框、语言切换、设置页失效问题。
- 增加内联 JS 语法回归测试。
- 补齐内置英文 UI 文本。
- 优化设置页为分组式布局。
- JSON UI 语言包输出的 `name` / `native_name` 使用目标语言显示名。

## P0：稳定性与排查能力

### 1. 浏览器级前端自检

**目标**：自动验证核心 UI 交互没有断。

**范围**：

- 启动本地 Web 服务。
- 打开首页。
- 点击源语言、目标语言、模型、主题、界面语言下拉框。
- 切换界面语言。
- 打开设置页。
- 验证设置页关键按钮存在且没有 JS 报错。

**实现建议**：

- 新增 `tests/browser/` 或 `scripts/ui_smoke_test.mjs`。
- 使用 Playwright。
- CI 或本地命令可运行：

```powershell
$env:PYTHONPATH="src"
python -m mc_mod_i18n serve --port 8765
node scripts/ui_smoke_test.mjs http://127.0.0.1:8765
```

**验收标准**：

- 任意核心下拉框打不开时测试失败。
- 页面出现未捕获 JS 异常时测试失败。
- 设置页不能打开时测试失败。

### 2. API 连接测试

**目标**：用户开始翻译前确认 BaseURL、Key、模型配置是否可用。

**范围**：

- 在高级 API 设置中增加“测试连接”按钮。
- 支持 OpenAI Compatible 和 Anthropic Compatible。
- 检查模型列表接口或最小 chat/messages 请求。
- 返回结构化错误：网络错误、401/403、404 路径错误、429 限流、模型不存在、响应格式异常。

**后端接口草案**：

```text
POST /api/test-provider
{
  "provider": "openai-compatible",
  "api_url": "https://api.openai.com/v1",
  "api_key": "...",
  "api_key_env": "OPENAI_API_KEY",
  "model": "gpt-4o-mini"
}
```

**返回草案**：

```json
{
  "ok": true,
  "provider": "openai-compatible",
  "model": "gpt-4o-mini",
  "latency_ms": 830,
  "message": "连接正常"
}
```

**验收标准**：

- Key 错误能明确提示认证失败。
- BaseURL 路径错误能明确提示接口地址不可用。
- 成功时展示延迟和实际使用的 provider/model。
- API Key 不写入日志明文。

## P1：任务闭环

### 3. 翻译前预检

**目标**：上传后先告诉用户即将处理什么，降低误操作。

**范围**：

- 扫描输入文件，不发起翻译。
- 显示识别到的源语言文件数量、目标语言是否已存在、预计输出文件名。
- 对 FTB Quests 显示识别到的 quests/lang 根目录。
- 对 JSON 显示 schema 类型：普通语言 JSON 或 UI 语言包 JSON。

**界面建议**：

- 在“开始生成”前展示一个预检摘要区域。
- 摘要包含警告等级：信息、警告、阻断。
- 阻断项示例：没有找到源语言、文件不是有效 JSON。

**验收标准**：

- 未开始翻译时不创建最终输出。
- 预检结果和实际任务处理入口复用同一套扫描逻辑。
- 用户切换输入文件后预检结果自动失效或重新计算。

### 4. 任务历史页

**目标**：生成后的结果可找回，便于继续下载、查看报告和排查日志。

**范围**：

- 新增“任务历史”视图。
- 列出最近任务：时间、输入类型、目标语言、provider、状态、成功/失败数量。
- 提供下载资源包、报告、API 调试日志、硬编码报告的入口。
- 支持按状态和输入类型筛选。

**存储建议**：

- 在 Web workdir 下维护 `jobs/index.jsonl`。
- 每个任务完成或失败时追加一条摘要。
- 不把 API Key 写入历史。

**验收标准**：

- 重启服务后仍能看到历史任务。
- 删除或缺失文件时，界面显示“文件已不存在”，不报 JS 错。
- 历史记录最多保留可配置数量，避免 workdir 无限膨胀。

### 5. 失败项重试

**目标**：API 或校验失败时，只重试失败项，不重复翻译全部内容。

**范围**：

- 结果页列出失败项。
- 支持“只重试失败项”。
- 复用原任务的 provider/model/locale/pack_format 等配置。
- 新结果合并回报告和输出文件。

**验收标准**：

- 失败项重试不会覆盖已成功译文。
- 重试后报告能显示原失败状态和最新状态。
- 任务取消后不会产生半写入的最终下载文件。

### 6. 输出预览和差异检查

**目标**：下载前让用户快速确认译文质量。

**范围**：

- 结果页增加“输出预览”。
- 支持按状态筛选：已翻译、已有翻译、跳过、校验失败、API 失败。
- 展示 key、原文、译文、文件名、失败原因。
- 对 JSON/UI 语言包显示 metadata 预览。

**验收标准**：

- 大量条目分页或虚拟列表，不一次性渲染全部 DOM。
- 搜索不会阻塞页面。
- 占位符校验失败项突出显示。

## P2：翻译资产管理

### 7. 术语表管理

**目标**：把零散 glossary 文件变成可维护资产。

**范围**：

- 设置页或独立页面增加术语表管理入口。
- 支持新增、编辑、删除术语。
- 支持导入导出 JSON。
- 显示与内置术语冲突的条目。
- 用户术语优先级高于内置术语。

**数据格式建议**：

```json
{
  "Copper": "铜",
  "Mana": "魔力",
  "Create": "机械动力"
}
```

**验收标准**：

- 保存后的术语表可被 Web 翻译任务使用。
- CLI 仍支持 `--glossary glossary.json`。
- 冲突项有明确提示，不静默覆盖。

### 8. UI 语言包检查器

**目标**：降低维护界面语言包的成本。

**范围**：

- 导入语言包时展示缺失 key、额外 key、占位符不匹配。
- 支持导出缺失 key 模板。
- 支持按内置语言补全缺失项。

**验收标准**：

- `{name}` 这类占位符缺失时阻止导入或给出强警告。
- 额外 key 不导致导入失败，但在检查结果中显示。
- 检查逻辑复用 `ui_i18n.py` 的内置 key 集合。

### 9. 翻译记忆

**目标**：复用历史译文，提高一致性并减少 API 成本。

**范围**：

- 按 source locale、target locale、source text、provider/model、术语表指纹存储译文。
- 命中翻译记忆时直接复用。
- 提供清理和导出入口。

**验收标准**：

- 改 provider/model/术语表后不会错误复用旧译文。
- 记忆命中数在结果摘要中可见。
- 清理操作有二次确认。

## P3：工作流增强

### 10. 配置预设

**目标**：保存常用 API 和输出配置，减少重复填写。

**范围**：

- 保存 provider、BaseURL、model、并发、重试、batch size、pack format、输出策略。
- 不保存 API Key 明文；可保存环境变量名。
- 支持导入导出预设。

**验收标准**：

- 切换预设后表单自动同步。
- 删除预设不影响当前任务。
- 预设 JSON 不包含敏感 Key。

### 11. 更完整的报告导出

**目标**：报告不仅能看，还能交付和复盘。

**范围**：

- 报告导出 CSV/JSON。
- 增加 provider、model、耗时、缓存命中、失败原因统计。
- 失败项可单独导出。

**验收标准**：

- CSV 可直接用 Excel 打开。
- JSON 保留机器可读字段。
- 报告中不暴露 API Key。

## 暂缓事项

这些功能暂不建议近期投入：

- 账号系统和云同步。
- 多用户权限。
- 在线插件市场。
- 自动发布到第三方平台。
- 大规模协同翻译工作流。

原因是当前项目更适合保持本地工具定位；上述能力会显著增加部署、权限、安全和维护复杂度。

## 推荐实施顺序

1. 浏览器级前端自检。
2. API 连接测试。
3. 翻译前预检。
4. 任务历史页。
5. 失败项重试。
6. 输出预览和差异检查。
7. 术语表管理。
8. UI 语言包检查器。
9. 翻译记忆。
10. 配置预设。
11. 报告导出增强。

## 测试策略

- **单元测试**：扫描、预检、任务历史索引、语言包检查、术语表合并。
- **HTML 契约测试**：关键控件、按钮 ID、data-i18n key、视图入口。
- **浏览器自检**：下拉框、设置页、语言切换、API 连接按钮、结果页切换。
- **端到端样例**：小型 JAR、FTB Quests fixture、普通 JSON、UI 语言包 JSON。
- **安全回归**：API Key 脱敏、日志脱敏、历史记录不保存明文 Key。

## 成功标准

- 常见 UI 断裂能在自检中提前发现。
- API 配置错误能在翻译前定位。
- 用户能在重启服务后找回近期任务。
- 大任务失败后不需要从头重跑。
- 术语、语言包、翻译记忆逐步沉淀为可复用资产。
