# deep-translator 集成方案（优化版）

## 1. 目标

为当前项目新增一个 **零配置、低门槛、尽力而为** 的翻译 provider：`deep-free`。

它的定位不是替代现有 AI provider，也不是承诺稳定可用的高质量在线翻译，而是：

- 让没有 API Key 的用户也能直接跑通流程
- 在支持语言范围内尽量给出句子级译文
- 在第三方免费源不可用时安全回退，不把任务整体搞挂

一句话定义：

> `deep-free` 是一个“可直接试用、失败可回退、质量不承诺”的免费在线翻译入口。

---

## 2. 为什么要做

当前项目已有 4 类翻译器：

- `openai-compatible`
- `anthropic-compatible`
- `glossary`
- `copy`

现状问题：

- `openai-compatible` / `anthropic-compatible` 需要 API Key，不适合零配置用户
- `glossary` 和 `copy` 可离线运行，但不能提供通用机器翻译能力
- 用户第一次使用时，往往只想先“跑通一次”，不一定愿意先准备 Key

因此需要一个额外 provider，满足：

- 不依赖用户自备 Key
- 能覆盖一部分主流 MC 语种
- 失败时能回退为原文，而不是让整批任务中断

---

## 3. 方案边界

本方案只定义 **MVP 第一阶段**，明确不追求一步到位。

### 3.1 本期做什么

- 新增 provider：`deep-free`
- 基于 `deep-translator` 接入两个免费句子级引擎：
  - `GoogleTranslator`
  - `MyMemoryTranslator`
- 提供引擎回退链：
  - `google -> mymemory -> copy`
- 支持当前项目的 `Translator` / `translate_batch_with_failures` 接口
- 支持 Minecraft locale 到翻译引擎 locale 的受控映射
- 对不支持 locale、安全失败和网络失败给出明确回退

### 3.2 本期不做什么

- 不接入 `LibreTranslator`
  - 它更适合后续“自托管在线翻译”单独立项
- 不接入任何需要 Key 的 deep-translator 引擎
- 不把 `deep-free` 设计成高并发 provider
- 不承诺 127 个 MC locale 都能翻译
- 不把第三方真实联网测试放进默认自动测试链路

---

## 4. 产品定位与用户提示

`deep-free` 在 UI 和文档里必须明确提示以下事实：

- 免费引擎依赖第三方公共服务
- 可用性受网络环境、地区、限流、服务端结构变化影响
- 不保证质量、速度和稳定性
- 适合快速试用、小规模任务、无 Key 场景
- 大批量、稳定生产使用仍建议用现有 AI provider

推荐 UI help 文案：

> 免费公共翻译源，适合零配置试用。可能受网络、限流和第三方服务变动影响；如需稳定大批量翻译，建议使用兼容 OpenAI / Anthropic 的 provider。

## 6. MVP 引擎策略

### 6.1 引擎选择

本期只接入：

- `google`
- `mymemory`

原因：

- 都支持句子级翻译
- 都无需用户提供 Key
- 都已经被 `deep-translator` 封装
- 组合后可以形成一个最小回退链

### 6.2 回退顺序

默认顺序：

```text
google -> mymemory -> copy
```

原因：

- Google 覆盖面通常更大
- MyMemory 可作为第二选择
- 两者都失败后，复制原文比整个任务报错更安全

### 6.3 为什么不在本期接入 Libre

虽然原始方案提到 `LibreTranslator`，但我建议本期去掉，原因是：

- 它的价值主要在“自托管可控性”，不是零配置
- 需要额外的 base URL、部署或公共镜像约束
- 会把当前简单目标扩成“在线公共源 + 自托管服务”双路线
- 这会让测试、UI、错误处理和文档复杂度显著上升

如果后续要做，可以单独作为：

- `deep-libre`
或
- `libre-selfhosted`

单独立项。

## 7. locale 支持策略

### 7.1 原则

不要承诺“127 个全部支持”。

更稳妥的做法是：

- 维护一份 **项目内受支持映射表**
- 只对映射表内能确定转换的 locale 提供 deep-free 翻译
- 对无法稳定映射或第三方明确不支持的 locale，直接走受控回退

### 7.2 映射数据结构

建议在 `deep_translator_adapter.py` 中定义：

```python
@dataclass(frozen=True)
class DeepFreeLocaleSupport:
    minecraft_locale: str
    google: str | None
    mymemory: str | None
    status: str  # "supported" | "fallback-copy"
    note: str = ""
```

目的：

- 映射关系是结构化数据，不是一长段难维护的文本
- 后续测试可以直接验证结构
- UI 或日志里可以显示 fallback 原因

### 7.3 locale 分类

分成三类即可：

1. **双引擎支持**
   - `google` 和 `mymemory` 都可用

2. **单引擎支持**
   - 只有其中一个引擎可用

3. **不支持**
   - 直接回退为原文

### 7.4 不支持 locale 的处理

对虚构语言、方言语言、无稳定 ISO 对应关系的 MC locale：

- 不尝试“猜测翻译”
- 不抛全局异常
- 默认返回原文
- 在失败信息里记录：`unsupported locale mapping`

## 8. 翻译器接口设计

### 8.1 类设计

```python
class DeepFreeTranslator(Translator):
    def __init__(
        self,
        source_locale: str,
        target_locale: str,
        engine_order: list[str] | None = None,
        unsupported_mode: str = "copy",
        request_timeout: float = 10.0,
    ) -> None:
        ...

    def translate_batch(
        self, items: list[TranslationItem]
    ) -> dict[str, str]:
        ...

    def translate_batch_with_failures(
        self, items: list[TranslationItem]
    ) -> tuple[dict[str, str], dict[str, str]]:
        ...
```

### 8.2 参数约束

- `engine_order`
  - 默认 `["google", "mymemory"]`
  - 本期只允许这两个值

- `unsupported_mode`
  - 本期建议只支持 `"copy"`
  - 不做 `"skip"` / `"raise"` 的用户可配行为

原因：

- 当前输出链路默认假设每条输入最终都能得到一个目标字符串
- `skip` 的语义会影响 pack / json / ftbquests 输出一致性
- `raise` 会让一个 locale 不支持直接放大成整批失败，不适合作为零配置入口

如果未来确实要支持：

- `skip` 和 `raise` 必须单独设计状态语义和 UI 表现

### 8.3 为什么这里不保留原方案的 `unsupported='copy'|'skip'|'raise'`

原始设计看起来灵活，但它把输出语义复杂度提前引进来了。

在当前项目里，第一阶段更重要的是：

- 让任务能跑通
- 错误可解释
- 报告可复盘

所以建议先只做一种明确策略：

```text
不支持或引擎全失败 -> copy 原文 + 记录失败原因
```

## 9. 状态语义

这是原方案最需要补强的部分。

### 9.1 现有项目里的关键语义

当前系统已有：

- `translated`
- `existing`
- `skipped`
- `failed`
- `api_failed`
- `incomplete`
- `jar_failed`

### 9.2 deep-free 的推荐映射

#### 场景 A：引擎成功返回译文

- 状态：`translated`

#### 场景 B：引擎不可用，但最终回退为原文

推荐：

- 结果里返回原文，避免任务中断
- 失败表里记录该项失败原因
- 状态仍记为 `api_failed`

原因：

- “看起来成功但实际没翻”最容易误导用户
- 如果直接记 `translated`，报告会失真
- `api_failed + target=source` 能更真实反映降级情况

#### 场景 C：source / target locale 本身不支持

推荐：

- 结果里返回原文
- 失败表记录原因：`unsupported locale`
- 状态记为 `api_failed`

这样用户在失败项队列里能清楚看到这类条目。

### 9.3 不建议新增 `unsupported_source` 这种状态

原方案写了 `unsupported_source`。[集成方案.md] 旧版这里的问题是：

- 当前项目已有状态体系中没有它
- 前端筛选、汇总、历史、报告导出都要同步认识新状态
- 引入单独状态的收益不高

更稳的做法：

- 继续复用 `api_failed`
- 在 `message` 中写清楚具体原因

## 10. 失败与回退行为

### 10.1 批量处理原则

`deep-free` 必须遵守现有批量模型：

- 单条失败不应阻断整批
- 一个引擎失败后应按顺序尝试下一个引擎
- 全部引擎失败后，该条进入失败集合

### 10.2 单条处理流程

建议逻辑：

```text
for item in items:
  1. 检查 source/target locale 是否可映射
  2. 依次尝试 google / mymemory
  3. 首个成功结果直接返回
  4. 全部失败 -> translations[item.id] = item.text
  5. failed[item.id] = 汇总错误原因
```

### 10.3 错误消息要求

不要只写：

- `translation failed`

建议写成：

- `google: timeout; mymemory: 429 rate limited`
- `unsupported target locale: lol_us`
- `source locale mapping missing: tlh_aa`

这样报告和 UI 才有排查价值。

## 11. UI 设计

### 11.1 provider 名称

建议：

- `Deep Translator（免费试用）`

比“免费”更准确，因为它强调试用性质。

### 11.2 provider help

建议展示为明显的说明文本，而不是普通 provider 一样的中性文案。

例如：

> 免费公共翻译源，无需 API Key。可能因网络或第三方限制失败；失败条目会回退为原文并在报告中标记。

### 11.3 高级参数联动

当选择 `deep-free` 时：

- 隐藏或弱化 API Key / API URL / model 输入
- 保留 timeout
- 在并发数旁边提示：
  - 推荐并发 `1`
- 在 batch size 旁边提示：
  - 推荐较小批次

### 11.4 结果页展示

结果页无需新增新 tab，但要确保：

- `api_failed` 条目能被筛出来
- message 中能看到是 locale 不支持还是第三方失败

## 13. 测试策略

原方案里“真实调用 Google / MyMemory”不适合作为默认单测。

这里建议拆成两层。

### 13.1 默认自动测试

文件：

- `tests/test_deep_translator_adapter.py`

只做稳定测试：

1. locale 映射结构完整
2. 已声明支持的 locale 至少映射到一个引擎代码
3. 不支持 locale 会走 copy 回退
4. Google 失败时可回退到 MyMemory
5. 两个引擎都失败时会返回原文并记录失败原因
6. `translate_batch_with_failures()` 返回长度与输入一致
7. 与现有 `core.translate_batch_with_failures()` 协作正常

这些测试应通过 mock / fake translator 完成，不依赖真实联网。

### 13.2 可选集成测试

单独写成：

- `tests/test_deep_translator_live.py`

并默认跳过，只有显式开启时才跑。

例如通过环境变量控制：

- `MC_MOD_I18N_LIVE_TRANSLATION_TESTS=1`

内容只验证最小事实：

- 免费源当前可访问
- 返回非空字符串

不要把这些测试视作 CI 阻断条件。

1. 新增 `deep_translator_adapter.py`

验收：

1. 写 mock 单测

验收：

- 默认测试稳定通过
- 可选 live 测试在联网时可人工跑

### 第三阶段：体验优化

1. 调整 `deep-free` 的高级参数默认值
2. 结果页强化 fallback 错误说明

## 16. 最终建议

如果要做，我建议按下面这个版本推进：

- provider 名：`deep-free`
- 引擎：`google + mymemory`
- locale 策略：受控映射，不承诺全量支持
- 回退策略：失败即复制原文并记 `api_failed`
- 测试策略：mock 单测为主，live 测试可选
- UI 定位：零配置试用入口，不是生产级主力 provider

这是一个能落地、能维护、也更符合当前代码结构的方案。
