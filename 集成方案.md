# `deep-translator` 集成方案（基于当前实现的更新版）

## 1. 目标

继续完善当前项目里的 `deep-free` provider，使它成为一个：

- 无需 API Key
- 面向零配置试用
- 失败可安全回退
- 与现有报告/失败体系兼容

的免费翻译入口。

这里的重点已经不是“要不要接入”，而是：

- 对齐 `deep-translator` 的实际上游能力
- 收口当前仓库里已经落地的实现
- 明确继续集成时还需要补的缺口

---

## 2. 上游事实核对（截至 2026-05-14）

结合 PyPI 页面与本地已安装包接口，`deep-translator` 当前对本项目最相关的事实如下：

- PyPI 当前最新版本仍为 `1.11.4`
- 发布时间为 `2023-06-28`
- 当前 `pyproject.toml` 已锁定为：
  - `deep-translator>=1.11,<2`
- 对本期最有用的免费引擎仍是：
  - `GoogleTranslator`
  - `MyMemoryTranslator`
- 这两个引擎都提供：
  - `translate()`
  - `translate_batch()`
  - `get_supported_languages()`
- PyPI 文档明确提到：
  - 可以复用 translator 实例，避免频繁重新实例化
  - 支持 `proxies`
  - `translate_sentences` 已不推荐，建议使用 `translate_batch`

这几个事实带来两个结论：

1. `deep-translator` 能继续用，但它的上游发布已经比较久，没有近期发版信号  
2. 本项目后续集成应尽量把它隔离在 adapter 层里，不要把第三方库细节散到核心流程

## 4. 当前实现的行为定义

### 4.1 provider 定位

`deep-free` 的定位仍然应该保持：

- 免费公共翻译源
- 适合零配置试用
- 不承诺稳定性
- 失败时优先保住任务完成，而不是追求严格报错

### 4.2 当前回退链

当前代码里的引擎顺序是：

```text
google -> mymemory -> copy
```

这条策略是合理的，建议保留。

### 4.3 当前失败语义

当前 adapter 的行为是：

- 成功翻译：返回译文
- 某引擎失败：继续尝试下一个引擎
- 全部失败：返回原文，并在 `failed` 映射里记录原因
- locale 不支持：直接返回原文，并记录 `unsupported ... locale`

这与项目“尽力而为、批量不中断”的目标一致。

## 5. 继续集成时需要补强的重点

下面这些不是“可以以后再看”的泛泛建议，而是当前实现和上游能力之间已经能看到的具体缺口。

### 5.1 locale 映射现在偏启发式，后续应改成“显式支持表”

当前 `resolve_deep_free_locale()` 的规则大致是：

- 特殊 locale 手写映射
- 明确不支持的 locale 走黑名单
- 其他形如 `xx_yy` 的 locale，默认退化为语言前缀，例如 `fr_ca -> fr`

这个策略对 Google 往往还能工作，但对 MyMemory 不够稳。

已确认的一个事实是：

- `GoogleTranslator` 可接受较宽松的代码形式
- `MyMemoryTranslator` 对语言代码更严格，常要求区域化代码，例如 `en-US`、`zh-CN`

因此建议下一步把 locale 支持改成：

1. 以引擎各自的 `get_supported_languages(as_dict=True)` 为基础
2. 建一份项目内的静态映射表
3. 对 Minecraft locale 做“显式白名单映射 + 少量人工覆盖”
4. 不再把所有未知 locale 都自动截断成语言前缀

建议保留的数据结构：

```python
@dataclass(frozen=True)
class DeepFreeLocaleSupport:
    minecraft_locale: str
    google: str | None
    mymemory: str | None
    status: str
    note: str = ""
```

这个结构已经存在，可以继续沿用。

### 5.2 `request_timeout` 目前名义上存在，实际上没有真正透传

当前 `DeepFreeTranslator.__init__()` 已接收：

```python
request_timeout: float = 10.0
```

但 `_build_engine()` 里实际构造引擎时，当前代码是：

```python
GoogleTranslator(source=source_locale, target=target_locale)
MyMemoryTranslator(source=source_locale, target=target_locale)
```

也就是说：

- timeout 现在只保存在 adapter 实例上
- 对真实 `deep-translator` 引擎没有形成稳定的运行时约束

这会导致 UI/CLI 上的超时参数对 `deep-free` 的用户预期不一致。

建议：

- 明确确认 `deep-translator` 是否支持直接超时透传
- 如果库本身不支持稳定 timeout 参数，就在方案文档和 UI 中写清“该 provider 的 timeout 只能控制外围流程，不能强保证单请求超时”
- 不要保留一个看起来可控、实际上不生效的假参数语义

### 5.3 当前实现是“逐条实例化 + 逐条 translate”，没有利用上游推荐用法

PyPI 文档明确建议：

- 复用 translator 实例
- 批量场景优先用 `translate_batch()`

而当前实现对每个 item 都会：

1. 新建引擎实例
2. 调一次 `translate(item.text)`

问题是：

- 性能偏差
- 更容易触发免费源限流
- 不符合上游推荐实践

建议下一步至少优化成：

- 每个 locale pair + engine 只实例化一次
- 同一批次复用实例
- 优先使用 `translate_batch()`
- 如果某个引擎批量失败，再降级为逐条或整体切到下一个引擎

推荐思路：

```text
按 engine 分批处理
  -> 复用 translator 实例
  -> translate_batch(items_text)
  -> 成功项直接收下
  -> 整批失败时再切换下一引擎
```

### 5.4 UI 里的“测试连接”逻辑目前不适配 `deep-free`

当前前端已经把 `deep-free` 放进了 `providerPresets`，并暴露了“测试连接”按钮流程。

但后端 `/api/test-provider` 依然是按 AI provider 的模型探测逻辑实现的，核心流程依赖：

- API Key
- models endpoint
- model 列表校验

这不适用于 `deep-free`。

因此这里现在存在一个体验不一致：

- UI 看起来能测
- 实际后端语义仍然按 OpenAI / Anthropic provider 处理

建议二选一：

1. 对 `deep-free` 隐藏“测试连接”按钮  
2. 为 `deep-free` 单独实现轻量测试：
   - 用固定文本做一次短翻译
   - 只校验“能否返回非空结果或合理失败信息”

我更推荐第 2 种，因为它更符合用户预期。

### 5.5 当前还没有 live 集成测试

现有测试基本都是 mock/fake 驱动，这对默认 CI 是对的，但仍有一块缺口：

- 没有联网验证当前第三方免费源是否仍能工作

建议补一个默认跳过的 live 测试文件，例如：

- `tests/test_deep_translator_live.py`

通过环境变量显式开启：

- `MC_MOD_I18N_LIVE_TRANSLATION_TESTS=1`

只验证最小事实：

- `google` 或 `mymemory` 当前可访问
- 返回结果非空
- 失败时能给出可读错误信息

不要把这类测试纳入默认阻断。

### 5.6 代理能力还没在本项目里暴露

PyPI 文档明确说明 `deep-translator` 支持 `proxies`。

当前项目里的 `deep-free` adapter 没有：

- 从 CLI 读取代理配置
- 从 UI 暴露代理字段
- 向底层 translator 透传 proxies

这不是 MVP 阻塞项，但如果面向国内网络环境继续完善，代理支持会明显提高可用性。

建议把它放到第二优先级，而不是现在立刻加。

## 6. 建议的继续推进方案

### 6.1 第一优先级：把现有实现收口到“可解释、可维护”

1. 保留 `deep-free` 作为 provider 名，不改对外语义
2. 保留 `google -> mymemory -> copy` 回退链
3. 把 locale 规则从“启发式猜测”升级为“显式映射表优先”
4. 明确 `request_timeout` 的真实能力边界
5. 修正 `deep-free` 的“测试连接”体验

### 6.2 第二优先级：利用上游批量能力优化实现

1. translator 实例复用
2. 优先使用 `translate_batch()`
3. 控制更小批次，降低免费源触发限流的概率

### 6.3 第三优先级：补充可选联网验证

1. 增加 `tests/test_deep_translator_live.py`
2. 文档里明确其用途与开启方式
3. 不将其纳入默认自动测试阻断

## 7. 修改文档后建议的实施清单

如果继续推进，我建议按这个顺序做：

1. 收紧 locale 映射
   - 基于 `get_supported_languages(as_dict=True)` 生成支持表
   - 保留项目侧特殊 locale 覆盖
2. 修 `deep-free` 连接测试
   - 单独实现 provider smoke test，或隐藏按钮
3. 优化 adapter 批处理
   - 实例复用
   - 优先 `translate_batch()`
4. 补 live test
   - 环境变量控制
   - 默认跳过
5. 最后再考虑代理支持

---

## 8. 验收标准

继续集成完成后，至少应满足：

- `deep-free` 在 UI、CLI、核心流程中的行为一致
- locale 不支持时能稳定回退，并给出可读原因
- 免费源失败不会拖垮整批任务
- “测试连接”不会再误走 AI provider 语义
- 默认测试稳定通过
- 可选 live test 可在联网环境下手工验证

## 9. 其他免费翻译库 / API 备选项（截至 2026-05-14）

下面这张表只列对当前项目真正有参考价值的免费候选，不追求“列全世界所有翻译服务”。

| 选项 | 形态 | 免费方式 | 优点 | 主要限制 | 适合本项目的程度 |
| --- | --- | --- | --- | --- | --- |
| [Argos Translate](https://github.com/argosopentech/argos-translate) | 本地 Python 库 / 离线模型 | 开源免费、离线可跑 | 不依赖第三方在线服务；适合做真正的离线 provider | 语言覆盖和质量通常不如主流云翻译；需要单独安装语言包 | 高，适合后续做 `argos-offline` |
| [LibreTranslate](https://docs.libretranslate.com/) | 在线 API / 可自托管 | 自托管免费；官方托管实例通常需要 key | 开源、接口简单、可完全自控；官方文档明确支持 self-hosted | 公共实例稳定性一般；自托管有部署成本 | 很高，适合后续做 `libre-selfhosted` |
| [MyMemory](https://mymemory.translated.net/doc/spec.php) | 公共在线 API | 匿名 `5000 chars/day`；带邮箱参数 `50000 chars/day`；白名单项目 `150000 chars/day` | 零配置、直接可用 | 配额很小；质量和稳定性有限；更适合试用而非大批量 | 已接入，适合作为回退源而非主力 |
| [Azure Translator](https://azure.microsoft.com/en-us/pricing/details/cognitive-services/translator/) | 官方云 API | `F0` 免费层，当前页面显示 `2 million characters` 免费 / 月 | 官方稳定，文档完整，额度对中小项目很实用 | 需要 Azure 账号和 key；不是“零配置” | 很高，适合做 `azure-translator` |
| [Google Cloud Translation](https://cloud.google.com/translate/pricing) | 官方云 API | 当前定价页写明前 `500,000` 字符 / 月免费（按月 credit 抵扣） | 官方能力强，语言覆盖广 | 需要云账号和计费配置；超额后直接计费 | 中高，适合做“稳定付费优先”方案 |
| [Amazon Translate](https://aws.amazon.com/translate/pricing/) | 官方云 API | 当前免费层为 `2M` 字符 / 月，持续 `12 个月` | 官方稳定，额度不小 | 免费期不是长期；AWS 接入复杂度更高 | 中，除非项目本身已用 AWS |
| [DeepL API](https://support.deepl.com/hc/en-us/articles/360021200939-DeepL-API-plans) | 官方云 API | 官方当前帮助中心显示：旧 `API Free` 不再可新购；现有 API 计划以 DeepL 当前产品页为准 | 质量通常较好 | 免费方案策略近期有变化，不适合作为“稳定免费接入”基线 | 中，适合单独评估，不建议现在作为免费主方案 |
| [googletrans](https://pypi.org/project/googletrans/) | 非官方 Python 库 | 免费 | 接入简单 | PyPI 明确标注为非官方实现；稳定性、封禁风险、长期可维护性都差 | 低，不建议主仓接入 |

### 9.1 选型建议

如果只从当前项目的实际需求出发，我建议这样排序：

1. 零配置免费试用：继续保留 `deep-free`，但把它收口得更稳
2. 真免费且可控：优先补 `LibreTranslate` 自托管路线
3. 真离线：再评估 `Argos Translate`
4. 官方稳定免费额度：优先评估 `Azure Translator`

### 9.2 为什么优先推荐 LibreTranslate / Argos / Azure

- `LibreTranslate`
  - 最接近“免费 + 可控 + 可作为正式 provider”
  - 不会像公共免费源那样完全受第三方页面结构和反爬策略影响
- `Argos Translate`
  - 是少数真正适合做“离线 provider”的候选
  - 如果项目以后想支持完全无网环境，这是最值得单独立项的方向
- `Azure Translator`
  - 虽然不是零配置，但免费额度够大，且比公共免费源稳定得多
  - 如果用户愿意配 key，它比 `deep-free` 更适合作为“低成本长期方案”

### 9.3 不建议现在作为主线推进的选项

- `googletrans`
  - 非官方实现，项目维护风险过高
- 单纯依赖公共免费网页源
  - 容易因为限流、网络、页面变更失效
- 当前形态下的 `DeepL API Free`
  - 官方计划近期有调整，免费策略不适合作为稳定基础假设

## 10. 最终建议

当前这条线不需要推翻重做。

更合适的方向是：

- 承认 `deep-free` 已经完成了 MVP 接入
- 用 `deep-translator` 的现有接口把它收口得更稳
- 明确它始终只是“免费试用入口”，不是生产级主力 provider

一句话总结：

> 接下来要做的不是“再接一个 deep-translator”，而是把已经接进来的 `deep-free` 做到行为清晰、映射可控、失败可解释、体验闭环。
