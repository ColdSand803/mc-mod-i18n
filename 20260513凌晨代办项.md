## 二、硬编码翻译生效问题

当前 `hardcoded-map.json` 只是一份“原文 -> 译文”的映射数据，不是资源包语言文件。Minecraft 资源包只能覆盖 `assets/<modid>/lang/*.json` 这类资源，无法直接替换 Java class 里写死的字符串。因此：

- 单独把 `hardcoded-map.json` 放进资源包不会生效。
- 当前工具已能扫描、分类、AI 翻译、校验并导出映射。
- 真正在游戏中生效需要一个运行时消费者读取映射并替换显示文本。

可行路线：

1. **补丁 Mod / Mixin 路线**
   - 做一个 companion patch mod。
   - 启动时读取 `config/mc-mod-i18n/hardcoded-map.json` 或资源包中的约定路径。
   - 在 Ponder、Tooltip、Screen、Config 注释等显示入口做原文查表替换。
   - 优点：能覆盖真正的硬编码显示文本。
   - 风险：需要按 MC 版本、Loader、目标 Mod API 维护 Mixin 入口。

2. **KubeJS / 脚本配置路线**
   - 将部分硬编码映射转换成 KubeJS、配置模板或目标 Mod 支持的脚本格式。
   - 优点：部署门槛较低。
   - 风险：只能覆盖脚本系统能接管的文本，不能通用替换所有 Java 写死字符串。

3. **仅报告辅助路线**
   - 保持当前工具只生成报告和映射，交给人工或专用补丁项目处理。
   - 优点：维护成本最低。
   - 风险：用户会继续问“导出后怎么用”，交付闭环不完整。

建议下一步优先做路线 1 的最小可行版本：

- 新建一个独立 patch mod 模板目录。
- 先支持 NeoForge 1.21.1。
- 只处理一个高价值入口，例如 Ponder 文本或 Screen tooltip。
- 约定读取 `config/mc-mod-i18n/hardcoded-map.json`。
- Web 导出完成后给出“放到 config/mc-mod-i18n/ 并安装补丁 Mod”的明确说明。
