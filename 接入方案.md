# 翻译 Provider 总接入方案

## 1. 目标

为当前项目建立一套可持续扩展的翻译 provider 体系，覆盖四条路线：

- `deep-free`
  - 零配置免费试用入口
- `LibreTranslate`
  - 开源在线 API / 自托管路线
- `Argos Translate`
  - 真离线本地翻译路线
- `Azure Translator`
  - 官方稳定免费额度路线

这四条路线不是同一类能力，不能一次混做。总方案的核心目标是：

- 统一 provider 接口和失败语义
- 统一 UI / CLI 的配置体验
- 让不同 provider 可以按阶段逐个接入
- 保持测试、报告和回退逻辑一致

---

## 2. 总体原则

### 2.1 统一接口，不统一实现细节

所有 provider 都应继续适配项目现有的翻译接口：

- `Translator.translate_batch()`
- `Translator.translate_batch_with_failures()`

但每种 provider 的内部实现不强行统一：

- `deep-free`
  - 适合“多免费源回退 + 尽力而为”
- `LibreTranslate`
  - 适合“单服务 endpoint + 可选 key”
- `Argos Translate`
  - 适合“本地模型 + 本地推理”
- `Azure Translator`
  - 适合“官方云 API + 明确配额与认证”

统一的是输入输出语义，不是内部调用方式。

### 2.2 统一失败语义

所有 provider 都应满足：

- 单条失败不拖垮整批
- 失败项返回可读原因
- 在可安全回退的场景下优先保住整体任务完成

建议保持当前项目语义：

- 成功翻译：正常返回译文
- 失败但可回退：返回原文，并在 `failed` 中登记原因
- 不支持 locale：返回原文，并登记 `unsupported locale`

### 2.3 统一配置分层

按配置性质分三层：

1. 通用配置
   - `source_locale`
   - `target_locale`
   - `api_timeout`
   - `api_batch_size`
   - `api_concurrency`
2. provider 基础配置
   - `api_url`
   - `api_key_env`
   - `model`
3. provider 专属配置
   - `deep-free` 的引擎顺序
   - `LibreTranslate` 的实例类型 / key
   - `Argos` 的模型包与安装状态
   - `Azure` 的 endpoint / region / key

通用配置尽量共享，专属配置通过 provider 分支处理，不强塞进同一套字段。

### 2.4 测试分层

所有新 provider 都按三层测试：

1. 纯单测
   - mock / fake 驱动
   - 默认 CI 必跑
2. 集成测试
   - 对项目内部流程做协同验证
3. 可选 live test
   - 需要联网或真实环境
   - 默认跳过

## 3. 分阶段路线图

### 第一阶段：`deep-free` 收口

目标：

- 把已经存在的 `deep-free` 从 MVP 接入提升到正式可维护状态

本阶段做：

- 收紧 locale 映射
- 优化引擎实例复用和批处理
- 修正 provider 测试逻辑
- 补 live test 骨架
- 同步文档和 README

当前执行策略：

- 本轮先完成 `deep-free` 的 provider smoke test、批处理优化和 live test 骨架
- locale 映射先从“高频主流 locale 更稳”做起，不在第一轮强求全量穷举

本阶段不做：

- 新增其他 provider

### 第二阶段：`LibreTranslate` provider

目标：

- 增加一个真正适合作为“免费但可控”正式 provider 的路线

本阶段做：

- 接入 `LibreTranslate` API
- 支持公共实例和自托管实例
- 支持有 key / 无 key 两种模式
- 补 provider smoke test

当前执行状态：

- 已完成最小 provider 闭环
- 当前默认 BaseURL 为 `http://127.0.0.1:5000`
- 优先按自托管实例场景设计，托管实例通过手动修改 BaseURL 使用

### 第三阶段：`Argos Translate` provider

目标：

- 增加离线翻译能力

本阶段做：

- 接入本地翻译库
- 增加模型可用性检查
- 明确缺失模型包时的提示和失败语义

当前执行状态：

- 已完成最小 provider 闭环
- 当前不负责在线下载模型包
- 优先覆盖“依赖缺失 / 模型缺失 / 已安装可翻译”三种本地状态

### 第四阶段：`Azure Translator` provider

目标：

- 增加一个官方稳定、适合长期使用的低成本 provider

本阶段做：

- 接入 Azure Translator API
- 支持 endpoint / region / key
- 支持连接测试和模型/服务可用性检查

## 4. 各 provider 定位

### 4.1 `deep-free`

定位：

- 零配置免费试用入口
- 失败可回退
- 不承诺稳定性和大批量能力

适合：

- 第一次跑通流程
- 无 key 用户
- 小规模试用

不适合：

- 大批量稳定生产翻译

### 4.2 `LibreTranslate`

定位：

- 免费、可控、可自托管的正式 provider

适合：

- 需要稳定免费方案
- 能接受自托管或固定实例配置

### 4.3 `Argos Translate`

定位：

- 真离线 provider

适合：

- 无网环境
- 不希望请求任何外部服务

不适合：

- 追求最高翻译质量

### 4.4 `Azure Translator`

定位：

- 官方稳定 provider

适合：

- 愿意配置 key
- 需要长期低成本稳定使用

---

## 5. 共性设计

### 5.1 locale 支持策略

不再追求“默认猜测一切 locale 都能翻”。

统一策略：

- 每个 provider 都维护自己的 locale 适配层
- 对可稳定映射的 locale 明确支持
- 对不支持 locale 直接受控回退
- 错误消息里明确写是 source 还是 target 不支持

### 5.2 provider 测试策略

“测试连接”不再假设所有 provider 都有：

- API Key
- `/models`
- `model` 概念

统一改成 provider-specific smoke test：

- `deep-free`
  - 短文本翻译测试
- `LibreTranslate`
  - endpoint 连通性 + 简短翻译
- `Argos`
  - 本地模型可用性检查
- `Azure`
  - 官方接口认证与短文本请求

### 5.3 批处理与并发

按 provider 区分：

- `deep-free`
  - 低并发、小批次、优先减少限流
- `LibreTranslate`
  - 中低并发、受实例能力约束
- `Argos`
  - 受本地 CPU / 内存约束
- `Azure`
  - 可较高并发，但仍应受配额和速率控制

### 5.4 UI 规则

当 provider 切换时：

- 只展示该 provider 需要的字段
- 隐藏无意义字段
- 在 help 文案里清楚写出定位和风险

例如：

- `deep-free`
  - 不显示 API key 必填语义
- `Argos`
  - 不显示 API URL
- `Azure`
  - 显示 endpoint / region / key

## 6. 第一阶段详细范围：`deep-free` 收口

### 6.1 当前现状

仓库内已经有：

- `src/mc_mod_i18n/deep_translator_adapter.py`
- `core.py` 中的 `deep-free` 分支
- provider preset / UI 文案
- 基础单测

所以第一阶段不是“从零接入”，而是“补强收口”。

### 6.2 第一阶段实施项

1. locale 映射收紧
   - 区分 `GoogleTranslator` 和 `MyMemoryTranslator` 的语言代码习惯
   - 降低启发式猜测比例
2. 批处理优化
   - translator 实例复用
   - 优先 `translate_batch()`
3. provider 测试修正
   - 不再按 AI provider 的 `/models` 语义测试
   - 改成 `deep-free` 自己的 smoke test
4. live test 骨架
   - 默认跳过
   - 联网时可人工验证

### 6.3 第一阶段验收标准

- `deep-free` 的“测试连接”可在无 API key 场景下工作
- `deep-free` 失败时仍回退原文且不拖垮整批
- 批量翻译时优先复用实例并支持批量调用
- locale 不支持时失败原因清晰
- 默认单测通过

---

## 7. 第二阶段详细范围：`LibreTranslate`

### 7.1 目标

提供一个比 `deep-free` 更可控、比 `Azure` 更低门槛的正式免费 provider。

### 7.2 设计要点

- 新增独立 adapter，例如：
  - `src/mc_mod_i18n/libretranslate_adapter.py`
- 支持：
  - `base_url`
  - 可选 `api_key`
  - source/target locale 映射
- 失败时遵循统一失败语义

当前已落地能力：

- provider preset / CLI / core / Web UI 已接通
- `translate_batch_with_failures()` 已支持数组批量请求
- provider-specific smoke test 已接通
- 默认不要求 API key

### 7.3 风险

- 公共实例可用性不稳定
- 自托管与公共实例能力可能不完全一致

---

## 8. 第三阶段详细范围：`Argos Translate`

### 8.1 目标

提供离线 provider。

### 8.2 设计要点

- 独立 adapter
- 本地模型包检测
- 缺失模型包时给出明确安装指引
- UI 中突出“离线、本地、首次配置成本更高”

当前已落地能力：

- provider preset / CLI / core / Web UI 已接通
- provider-specific smoke test 已接通
- 已支持本地 backend 注入，便于 mock/fake 测试

### 8.3 风险

- 模型包管理复杂
- 语言覆盖有限
- 批量性能可能一般

---

## 9. 第四阶段详细范围：`Azure Translator`

### 9.1 目标

提供稳定、官方、低成本 provider。

### 9.2 设计要点

- 独立 adapter
- endpoint / region / key 配置
- provider smoke test
- locale 支持表
- Web UI / CLI / 核心流程统一透传 `api_region`

### 9.3 当前落地状态

- 已新增 `src/mc_mod_i18n/azure_translator_adapter.py`
- 已接入 `translator.py` provider preset、`cli.py` 参数、`core.py` 实例化分支
- 已在 `web.py` 中补齐 provider 选项、Region 输入、Azure 专用 smoke test 和配置预设透传
- 已补自动测试：adapter、provider connection、UI 合约

### 9.4 风险

- 不是零配置
- 需要处理官方配额与认证错误

## 12. 推荐实施顺序

推荐严格按以下顺序推进：

1. `deep-free` 收口
2. `LibreTranslate`
3. `Argos Translate`
4. `Azure Translator`

理由：

- `deep-free` 已有代码，收益最快
- `LibreTranslate` 最接近“免费且可控”的正式路线
- `Argos` 是独立的离线路线，复杂度不同
- `Azure` 最稳定，但不是零配置，适合最后补齐

---

## 13. 最终建议

这个总方案的关键不是“把 4 个 provider 都接上”，而是：

- 先把每一条路线的定位说清
- 再按阶段逐个做成完整闭环

一句话结论：

> 以 `deep-free` 收口为第一期，按 `LibreTranslate -> Argos Translate -> Azure Translator` 的顺序逐步接入，统一接口与失败语义，但保留每个 provider 自己的实现边界。
