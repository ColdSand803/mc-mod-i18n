# mc-mod-i18n 桌面版本方案

## 目标

将 mc-mod-i18n 从“需要命令行启动的本地 Web UI 工具”收敛为“用户双击即可使用的单窗口桌面应用”。

桌面版第一阶段不重做现有 Web UI，不改变核心翻译能力，而是把现有 UI 包进桌面窗口，并补齐应用数据目录、打包入口和后续安装器基础。

最终用户体验：

```text
双击 mc-mod-i18n
-> 打开单窗口桌面应用
-> 使用现有 Web UI 完成翻译、查看任务历史、管理输出
-> 关闭窗口即退出应用和本地服务
```

## 产品定位

桌面版是项目的主产品形态。

CLI 保留，但定位调整为：

- 开发调试入口
- 自动化/测试入口
- 高级用户的备用入口
- UI 后端能力的薄封装

面向普通用户的文档、打包、快捷方式、安装器都应以桌面 UI 为主。

## 窗口形态

桌面应用采用单窗口工具形态：

- 无浏览器地址栏
- 无托盘
- 无后台常驻
- 无复杂菜单栏
- 关闭窗口即停止本地服务并退出
- 默认打开现有 Web UI
- 端口冲突时自动选择空闲端口
- 启动失败时给出清晰错误提示

第一版不设计新的首页/工作台。先沿用现有 Web UI，避免把“桌面化”与“UI 信息架构重构”混在一起。

## 推荐技术路线

采用：

```text
Python core + 现有 Web UI + pywebview 桌面壳 + PyInstaller 打包 + 安装器
```

理由：

- 当前项目核心能力在 Python 侧，迁移成本最低。
- 现有 Web UI 已能承载主要交互，不需要重写前端。
- pywebview 可以提供原生窗口体验，同时继续复用本地 HTTP 服务。
- PyInstaller 已经在项目中使用，现有打包链路可以延续。
- 后续安装器只负责安装程序本体、创建快捷方式和卸载，不需要承载复杂业务逻辑。

暂不推荐 Electron 或 Tauri：

- Electron 会引入较重的 Node/Chromium 打包体系。
- Tauri 更适合前端独立、后端通过命令或 sidecar 接入的结构。
- 当前阶段的主要目标是纠正启动体验，不是重写应用架构。

## 启动架构

新增桌面启动入口，例如：

```text
mc_mod_i18n.desktop
```

或 CLI 子命令：

```powershell
python -m mc_mod_i18n desktop
```

桌面入口负责：

1. 解析桌面模式配置。
2. 初始化应用数据目录。
3. 选择空闲本地端口。
4. 在后台线程启动现有 Web 服务。
5. 用 pywebview 创建桌面窗口。
6. 加载 `http://127.0.0.1:<port>`。
7. 窗口关闭时停止服务并退出进程。

现有 `serve` 命令继续保留，用于开发模式和浏览器调试。

## 数据目录设计

桌面模式不应继续默认写入项目目录下的 `.ui_runs`。

推荐默认目录：

```text
%LOCALAPPDATA%\mc-mod-i18n\
```

建议结构：

```text
%LOCALAPPDATA%\mc-mod-i18n\
├─ app.db
├─ jobs\
├─ cache\
├─ outputs\
├─ extensions\
├─ logs\
└─ settings.json
```

各目录职责：

- `app.db`：后续 SQLite 数据库。
- `jobs/`：任务运行目录、临时文件、任务产物。
- `cache/`：翻译缓存、checkpoint、FTB Quests 缓存等。
- `outputs/`：用户生成的资源包、报告、补丁包等可下载产物。
- `extensions/`：语言拓展包、UI 语言包、术语包等外部扩展。
- `logs/`：应用日志、启动日志、API debug 日志。
- `settings.json`：第一阶段保留轻量设置；后续可逐步迁入 SQLite。

开发模式仍可使用：

```text
.ui_runs\
```

这样可以避免开发数据污染正式用户数据。

## 安装器职责

安装器只负责程序级工作：

- 选择安装路径
- 安装应用文件
- 创建桌面快捷方式
- 创建开始菜单快捷方式
- 注册卸载项
- 可选：安装后启动应用

安装器不负责：

- 写入大量默认缓存
- 绑定用户数据目录
- 迁移历史数据
- 初始化复杂数据库内容

用户数据应由应用首次启动时初始化。这样升级、卸载和重装都更安全。

## SQLite 迁移策略

SQLite 不作为第一阶段桌面化的阻塞项。

推荐原则：

- SQLite 存索引、状态、元数据。
- 大文件继续存在文件系统中。
- 不把资源包、HTML 报告、ZIP、日志等二进制或大文本全部塞入数据库。

优先迁移顺序：

1. 任务历史
   - 当前为 `jobs/index.jsonl` 和 `jobs/settings.json`。
   - 适合迁为 `jobs`、`job_outputs`、`job_summaries` 等表。
   - UI 的查询、删除、清理、筛选会更稳定。

2. 应用设置
   - provider 默认值
   - 最近使用路径
   - 目标语言
   - 模型配置
   - 历史保留数量

3. 缓存索引
   - source hash
   - config hash
   - provider/model
   - cache path
   - last used time
   - hit count

4. 语言拓展包索引
   - 包名
   - 版本
   - locale
   - 安装路径
   - 启用状态
   - 校验结果

5. 后续高级能力
   - 任务搜索
   - 缓存清理策略
   - 拓展包升级记录
   - provider 使用统计

迁移时需要兼容旧数据：

- 若发现 `.ui_runs/jobs/index.jsonl`，可提供导入能力。
- 若发现旧版桌面数据目录中的 JSONL，也应自动迁移或提示迁移。
- 迁移完成后保留原文件备份，避免数据损坏不可恢复。

## 打包方案

现有项目已有 PyInstaller 相关文件：

```text
mc-mod-i18n.spec
scripts/build_exe.ps1
```

桌面版打包应调整为：

- 默认 exe 启动桌面模式。
- CLI 能力通过子命令保留。
- PyInstaller bundle 包含 Web UI 所需静态资源。
- pywebview 依赖纳入打包。
- 启动失败日志写入 `%LOCALAPPDATA%\mc-mod-i18n\logs`。

可考虑两个 exe：

```text
mc-mod-i18n.exe          # 桌面主入口
mc-mod-i18n-cli.exe      # 可选 CLI 入口
```

第一阶段也可以只保留一个 exe：

```powershell
mc-mod-i18n.exe          # 默认 desktop
mc-mod-i18n.exe serve    # 开发/高级入口
mc-mod-i18n.exe translate ...
```

## 分阶段计划

### P0：桌面壳

目标：双击打开单窗口应用。

内容：

- 新增 desktop 启动入口。
- 集成 pywebview。
- 自动选择空闲端口。
- 后台启动现有 Web 服务。
- 窗口关闭时退出服务。
- 继续加载现有 Web UI。

验收：

- 用户无需打开命令行。
- 用户无需手动打开浏览器。
- 关闭窗口后进程退出。
- 端口占用时仍能启动。

### P1：应用数据目录

目标：桌面模式使用正式用户数据目录。

内容：

- 新增应用数据目录解析。
- 桌面模式默认使用 `%LOCALAPPDATA%\mc-mod-i18n`。
- `serve` 开发模式继续默认 `.ui_runs`。
- 日志、任务、输出、缓存有稳定路径。

验收：

- 桌面版不在项目根目录写运行数据。
- 任务历史和输出在重启后保留。
- 开发模式不污染正式数据目录。

### P2：打包桌面 exe

目标：生成可直接运行的桌面版 exe。

内容：

- 更新 PyInstaller spec。
- 更新构建脚本。
- 默认启动 desktop。
- 验证静态资源、图标、模板、帮助文档都被打包。

验收：

- `dist/mc-mod-i18n/mc-mod-i18n.exe` 可双击启动。
- UI 正常加载。
- 翻译流程可跑通。
- 任务历史可记录。

### P3：安装器

目标：提供标准 Windows 安装体验。

内容：

- 选择 Inno Setup、NSIS 或 WiX。
- 支持选择安装路径。
- 创建桌面快捷方式。
- 创建开始菜单快捷方式。
- 支持卸载。

验收：

- 用户可通过安装包安装应用。
- 安装后可从桌面或开始菜单启动。
- 卸载不误删用户数据，除非用户明确选择清理。

### P4：SQLite 任务历史

目标：用 SQLite 替代任务历史 JSONL。

内容：

- 设计 `app.db` 初始 schema。
- 实现迁移逻辑。
- 保持现有 API 返回结构兼容。
- 更新任务历史读写测试。

验收：

- 旧任务历史可迁移。
- UI 行为不回退。
- 删除、清理、读取历史稳定。

### P5：缓存和拓展包索引

目标：逐步把缓存、语言拓展包等变成可管理资源。

内容：

- 缓存索引入库。
- 语言拓展包 manifest 入库。
- UI 增加管理能力。
- 支持清理无效缓存和禁用拓展包。

验收：

- 用户能查看缓存占用。
- 用户能清理缓存。
- 用户能查看和管理语言拓展包。

## 风险与注意事项

### 服务生命周期

桌面窗口关闭后，本地 HTTP 服务必须退出。不能留下后台进程。

### 端口管理

桌面模式不应固定依赖 `8765`。固定端口适合开发模式，不适合桌面应用。

### 路径兼容

Windows 路径可能包含中文、空格和特殊字符。所有文件操作应使用 `Path`，避免手写字符串拼接。

### 数据安全

API Key、Authorization header、debug log 都需要继续脱敏。SQLite 迁移时不能把敏感字段直接写入历史记录。

### 升级兼容

安装路径和用户数据路径应分离。升级应用不应覆盖用户历史、缓存、输出和扩展包。

### Web UI 资源

当前 Web UI 写在 Python 文件中，后续如果 UI 继续膨胀，可以再考虑拆分静态资源，但这不应阻塞桌面版第一阶段。

## 推荐结论

推荐立即推进桌面版，但采用渐进式路线：

```text
先桌面壳
再应用数据目录
再打包 exe
再安装器
最后 SQLite 化任务历史和资源索引
```

第一版的目标不是“重做一个桌面软件”，而是让现有 Web UI 获得正确的产品入口：用户双击一个窗口就能使用。
