# mc-mod-i18n 补充方案

> 范围：基于当前仓库代码状态梳理，不重复规划已经落地的能力。目标是把工具从“可用的本地翻译器”补成“可验证、可复用、可打包交付、可长期维护”的 Minecraft Mod 汉化工具。

## 1. 当前已具备的基础

当前项目已经不是单纯 MVP，下面能力已经在代码里存在：

- CLI 入口：`translate` / `serve`，支持单 JAR、目录批量 JAR。
- Web UI：本地上传、任务进度、取消任务、失败项重试、硬编码映射工作台。
- 翻译器：`copy`、`glossary`、OpenAI 兼容接口，以及 OpenAI、DeepSeek、Moonshot、DashScope、智谱、硅基流动、Gemini、Groq、MiMo 等预设。
- JAR 处理：支持 `META-INF/jarjar/`、`META-INF/libraries/`、`META-INF/jars/` 嵌套 JAR 扫描。
- 语言文件：支持 JSON / legacy `.lang`，JSON 值支持字符串和 dict 富文本结构的保留。
- 校验：占位符、花括号变量、Minecraft `§` 格式代码、换行、前后空白、目标 key 完整性检查。
- 稳定性：API 重试、429 退避协调、JSONL debug 日志、checkpoint、Web 共享缓存、pack_format 版本映射。
- 报告：语言报告按 JAR 折叠加载，错误聚合；硬编码扫描报告支持分类和搜索。
- 打包：已有 PyInstaller spec 和 `scripts/build_exe.ps1`。

因此后续补充重点不再是“能不能跑”，而是“跑得是否可信、结果是否可复用、交付是否稳定”。

## 2. P0：必须优先补齐

### 2.1 自动化测试体系

**问题**

当前仓库没有 `tests/`，`pyproject.toml` 也没有 dev/test 依赖。项目已经包含 JAR 解析、并发翻译、缓存、报告、Web API、PyInstaller 打包等高风险逻辑，没有回归测试会导致每次优化都很容易改坏旧能力。

**补充内容**

- 新增 `tests/fixtures/`，用测试代码动态生成最小 JAR，不提交真实大型 Mod。
- 新增核心单元测试：
  - `lang.py`：JSON、legacy `.lang`、富文本 dict、`\n`、转义等解析与渲染。
  - `validator.py`：缺失/多余占位符、颜色代码、换行、空译文、空白不一致。
  - `pack.py`：pack_format 版本映射、checkpoint 保存/加载、资源包结构。
  - `hardcoded.py`：嵌套 JAR 递归、常量池文本过滤、分类规则。
  - `core.py` / `cli.py`：单 JAR、目录批量、skip translated、resume。
  - `translator.py`：URL 归一化、流式/非流式响应解析、非 JSON 返回、缺失 id、重试日志。
- 新增轻量集成测试：
  - 用 `copy` provider 跑完整 CLI，验证 zip 和 report 输出。
  - 用本地 fake HTTP server 模拟 OpenAI 兼容接口，验证批量、失败、重试和 debug log。
- 在 `pyproject.toml` 增加可选 dev 依赖，例如 `pytest`，并固定最小 Python 版本。

**验收标准**

- `python -m pytest` 可在无网络环境下通过。
- 测试生成的 fixture JAR 都在临时目录内创建，不依赖 `测试mod/`。
- 每次修改核心翻译、缓存、报告、打包逻辑前先有对应失败测试。

### 2.2 并发翻译失败状态隔离

**问题**

`OpenAICompatibleTranslator.translate_batch()` 会把失败项写到实例字段 `self.failed_items`。CLI 和 Web 的多 JAR 并发会共享同一个 translator 实例；多个线程同时调用 `translate_batch()` 时，`self.failed_items` 可能被其他 JAR 的结果覆盖，导致失败条目误判、漏报或写入错误回退。

**补充内容**

- 把失败项从 translator 实例状态中移出，改成随本次调用返回。
- 可选接口：
  - 保守做法：新增 `translate_batch_with_failures()`，返回 `(translations, failed_items)`。
  - 兼容做法：保留 `translate_batch()`，内部调用新接口，只给旧调用方返回 translations。
- `core.process_zip()` 只读取本次调用的 `failed_items`，不再读取共享实例字段。
- 多线程写 debug log 保留锁；失败项 dict 使用当前调用局部变量，不跨 JAR 共享。

**验收标准**

- 用 fake provider 并发处理 3 个 JAR，其中 1 个 JAR 故意失败，失败只出现在对应 JAR 报告里。
- 同一 provider `--api-concurrency > 1` 时，报告数量、状态和输出资源包稳定一致。

### 2.3 checkpoint / cache 配置指纹

**问题**

当前 checkpoint 主要依赖 source hash。CLI `--resume` 如果换了 provider、model、api_url、glossary、prompt 或内置术语表，源文件没变时仍可能复用旧译文。Web 共享缓存已经包含部分参数，但缺少缓存 schema、工具版本、prompt 版本、内置 glossary 版本等信息。

**补充内容**

- 定义 `translation_config_hash`，至少包含：
  - source locale / target locale
  - provider / model / api_url
  - overwrite_existing / skip_translated
  - pack_format
  - 外部 glossary 文件 hash
  - 内置 glossary/phrase 表 hash
  - prompt version
  - cache schema version
- `save_checkpoint()` 写入 `source_hash` 和 `translation_config_hash`。
- `load_checkpoint()` 或调用方必须同时校验 source hash 与 config hash。
- Web `shared_cache_scope_dir()` 改用同一套指纹逻辑，避免 CLI/Web 行为不一致。

**验收标准**

- 同一 JAR 使用 `copy` 跑完后，再用 `glossary` 或 AI provider 跑，不会复用 copy 结果。
- 改外部 glossary 后再次运行，会重新翻译或重新命中新的缓存分区。
- 改 prompt version 后旧缓存不会被误用。

### 2.4 离线和 exe 打包资源

**问题**

Web UI 目前引用 jsDelivr Remix Icon 和 Google Fonts；打包 exe 后在离线环境会丢图标/字体。同时 PyInstaller spec 的 `datas=[]`，`co1dsand_logo.png`、`ui优化方案/logo/minecraft.svg` 没有明确打进产物，打包后的 UI 资源路径不稳定。

**补充内容**

- 把 UI 必需图标和字体改为本地资源，或提供可靠 fallback。
- PyInstaller spec 增加 `datas`：
  - `co1dsand_logo.png`
  - `ui优化方案/logo/minecraft.svg`
  - 如果保留外部字体，则至少保证无网络时 UI 不空白、不错位。
- Web 资源读取增加 PyInstaller 运行时路径兼容函数。
- README 补充“离线使用”和“exe 资源包含范围”。

**验收标准**

- 断网后运行 `mc-mod-i18n.exe serve`，页面图标、布局和资源包 `pack.png` 正常。
- `dist/mc-mod-i18n/` 可复制到另一台 Windows 机器运行，不依赖源码目录。

### 2.5 dry-run / 预检模式

**问题**

现在用户需要真正开始翻译后才知道会处理多少 JAR、多少条目、多少已有译文会跳过、是否命中缓存、预计会发多少 API 请求。

**补充内容**

- CLI 增加 `--dry-run`：
  - 扫描 JAR 和 nested JAR。
  - 统计源语言文件数、目标语言文件数、待翻译条目数、已有译文数、预检 warning/error。
  - 计算预计 API batch 数，但不调用 provider，不写资源包。
- Web UI 在开始翻译前增加“扫描预览”步骤：
  - 展示 JAR 数、条目数、风险数、缓存可复用数量。
  - 用户确认后再开始实际翻译。

**验收标准**

- dry-run 不创建资源包，不写入新的 checkpoint。
- dry-run 输出的待翻译条目数与实际翻译报告中的 `translated + failed + api_failed` 数量一致。

## 3. P1：提升日常效率

### 3.1 术语库管理

**问题**

当前只支持内置 glossary 和单次上传/传参的外部 `glossary.json`，术语不会自动沉淀，多个词表也没有冲突来源。

**补充内容**

- 增加项目级 `.mc-i18n/glossary.json`。
- 增加全局级 `%USERPROFILE%/.mc-i18n/glossary.json`。
- 合并优先级：临时上传/CLI 参数 > 项目级 > 全局级 > 内置词表。
- 冲突项在预检或日志中列出来源。
- Web UI 支持查看已加载术语来源、导入、导出。

**验收标准**

- 同一词条在项目级覆盖全局级。
- 冲突不会静默覆盖，报告中能看到最终生效来源。

### 3.2 翻译记忆 TM

**问题**

当前缓存粒度偏 JAR，不能把“一个英文短句的好译文”跨 Mod 复用。用户重复翻译多个 Mod 包时，仍会对相同文本重复调用 AI。

**补充内容**

- 新增本地 TM 文件：`.mc-i18n/translation-memory.jsonl` 或 SQLite。
- key 使用 source locale、target locale、source text、provider/model、术语表指纹。
- 命中策略：
  - 精确 source text 命中直接复用。
  - 同 key + 同 source text 优先级最高。
  - 相似匹配先只做离线建议，不自动写入，避免误译。
- 成功通过 validator 的译文才写入 TM。

**验收标准**

- 两个不同 JAR 包含同一英文文本时，第二个 JAR 不再调用 AI。
- 校验失败或 api_failed 的条目不会进入 TM。

### 3.3 Web 任务历史持久化

**问题**

Web 的 `jobs` 当前在内存中，服务重启后任务列表、状态和结果索引消失。`.ui_runs` 里有文件，但缺少稳定索引。

**补充内容**

- 新增 `.ui_runs/jobs.json` 或 `.ui_runs/jobs.sqlite`。
- 持久化 job_id、上传文件名、provider、model、状态、开始/结束时间、输出路径、摘要统计、失败数量。
- Web UI 增加历史任务列表：
  - 可打开报告。
  - 可重新下载产物。
  - 可基于旧参数重新运行。
  - 可清理历史和对应文件。

**验收标准**

- 重启 `serve` 后，已完成任务仍能在 UI 中看到并打开报告。
- 清理任务只删除 `.ui_runs` 内对应目录，不影响项目其他文件。

### 3.4 硬编码汉化闭环

**问题**

硬编码扫描和 `hardcoded-map.json` 导出已经具备，但还没有真正生成可用补丁 Mod 或可执行的运行时替换方案。

**补充内容**

- 明确第一版只支持“生成补丁 Mod 模板”，不直接修改原 Mod。
- 输出内容：
  - `hardcoded-map.json`
  - 补丁 Mod 工程骨架或资源模板
  - 使用说明
- 对不同类别分层处理：
  - `ponder` / `ui_literal` 优先。
  - `config_comment` 单独生成配置注释模板。
  - `unknown_literal` 默认不自动打包，必须人工确认。

**验收标准**

- 用户从 Web UI 导出 map 后，可以一键生成一个补丁工程目录。
- 高风险 unknown 文本不会被默认写入补丁。

### 3.5 JSON 富文本深度处理

**问题**

当前 JSON 值为 dict 时会保留结构并翻译顶层 `text` 字段，但 Minecraft 文本组件还可能在 `extra`、`with`、数组、嵌套对象中包含文本。

**补充内容**

- 新增文本组件遍历器：
  - 收集所有可翻译 `text` 字段。
  - 保留颜色、事件、样式、translate key、extra 等结构。
  - 翻译后按原路径写回。
- 对 `translate` 组件默认不翻译 key，只处理明确的展示文本字段。

**验收标准**

- 嵌套 `{"text":"A","extra":[{"text":"B"}]}` 输出后结构不变，A/B 均可翻译。
- 含 `clickEvent` / `hoverEvent` 的对象不丢字段。

## 4. P2：交付和维护能力

### 4.1 拆分超大 `web.py`

**问题**

`web.py` 同时包含 HTTP handler、任务调度、业务逻辑、HTML、CSS、JS，文件过大，后续维护成本高。

**补充内容**

- 保持“无前端构建工具链”的前提下拆分：
  - `web_server.py`：HTTP handler、路由、文件下载。
  - `jobs.py`：任务状态、取消、历史持久化。
  - `web_assets.py` 或 `templates/`：HTML/CSS/JS 字符串或静态文件。
  - `web_api.py`：translate、retry、hardcoded API 的参数解析。
- 拆分前先补测试，避免纯重构引入行为变化。

**验收标准**

- `python -m mc_mod_i18n serve` 行为不变。
- Web UI 的上传、翻译、重试、硬编码导入导出都通过回归测试或手工验收清单。

### 4.2 配置预设

**补充内容**

- 保存常用 provider、api_url、model、api_key_env、batch_size、concurrency、pack_format、术语表路径。
- CLI 支持 `--preset name`。
- Web UI 支持选择/保存/删除 preset。
- API Key 只保存环境变量名，不保存明文 key。

**验收标准**

- 选择 preset 后 CLI/Web 参数一致。
- 导出的 preset 文件不包含明文 API Key。

### 4.3 文档和发布流程

**补充内容**

- README 增加：
  - 快速开始。
  - CLI 参数完整表。
  - Web UI 使用流程。
  - 常见 provider 配置。
  - pack_format 说明。
  - 缓存、日志和隐私说明。
  - Windows exe 离线使用说明。
- 增加 `CHANGELOG.md`。
- 增加发布检查清单：
  - 测试通过。
  - exe 打包通过。
  - 样例 JAR 验证通过。
  - 断网 UI 验证通过。

**验收标准**

- 新用户只读 README 就能完成一次 copy/glossary 翻译。
- 发布前有固定命令清单，不靠临时记忆。

## 6. 不建议现在做的事

- 不建议引入大型前端框架。当前 UI 是本地工具，先把 `web.py` 拆清楚、资源本地化、任务持久化，比引入构建链更实际。
- 不建议直接修改原 Mod JAR。汉化产物应继续以资源包或补丁 Mod 形式输出，降低破坏原文件和授权风险。
- 不建议把 API Key 写入 preset 或 job 历史。只保存环境变量名和 provider 配置。
- 不建议在没有测试前继续大改并发和缓存。当前最容易出隐性问题的就是并发状态、缓存复用和报告一致性。

## 7. 最小可落地版本

如果只投入一轮开发，建议只做下面 5 件：

1. 加 pytest 测试框架和核心 fixture。
2. 修复 translator 并发失败状态隔离。
3. 给 checkpoint/cache 加配置指纹。
4. 加 CLI `--dry-run`。
5. 修复 exe 离线资源打包。

这 5 项完成后，项目的可靠性、可交付性和后续迭代速度会明显提升。
