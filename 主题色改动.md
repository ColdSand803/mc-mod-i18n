# 主题色改动计划

## 背景

本方案基于 `F:\dev\SafeVault\ui主题配色.md` 中提取的主题色统计，并对照当前项目 `src/mc_mod_i18n/web.py` 的内嵌 UI 实现评估。

SafeVault 的主题系统不是简单的主色替换。它包含 30 个主题入口，其中 `system` 是虚拟入口，真实 Web 主题为 29 个。每个真实主题提供一组核心色：`Accent`、`Accent Strong`、`Signal`、`Danger`、`BG`、`BG Deep`、`Ink`、`Muted`、`Surface`、`Surface Dark`。当前项目已经有浅色、暗色两套 CSS token，但变量粒度更细，包含表单、卡片、状态框、滚动条、下拉框、checkbox、loading、阴影等派生变量。

结论：SafeVault 主题可以迁移，但不建议直接复制色值覆盖当前变量。更稳妥的做法是先建立“SafeVault 核心 token -> 当前 UI 派生 token”的映射层，再按主题分组接入。

## 当前项目现状

- 主题代码集中在 `src/mc_mod_i18n/web.py` 的 `INDEX_HTML` 字符串中。
- CSS 当前只有 `:root` 浅色主题和 `:root[data-theme="dark"]` 暗色主题。
- JS 当前只有 `themeModes = ['auto', 'dark', 'light']`，`auto` 会根据系统明暗解析成 `dark` 或 `light`。
- 主题状态存储在 `localStorage` 的 `mc-mod-i18n-theme`。
- 顶部主题按钮现在是三态循环，不适合直接扩展为 30 个主题循环。
- 当前 CSS 已经大量使用语义变量，例如 `--accent`、`--accent-2`、`--success`、`--danger`、`--bg`、`--panel`、`--field-bg`、`--card-surface`、`--status-success-bg`。
- 仍有部分固定色需要清理，例如 `#fff`、部分 metric 色条、progress 色条、少量固定浅蓝背景和滚动条色。

## 目标

- 保留当前 UI 的信息密度和工作台风格，不做营销页式重设计。
- 接入 SafeVault 主题色，使用户可以在设置中选择主题。
- 保留 `auto/light/dark` 的已有行为和本地存储兼容性。
- 所有主题下保证表单、下拉框、表格、状态提示、loading、设置弹窗可读。
- 高饱和主题不影响状态语义，例如成功、警告、错误仍能被识别。

## 非目标

- 不引入新前端框架。
- 不重构页面布局。
- 不迁移 SafeVault 的 Android / 扩展端逻辑，只吸收 Web 主题 token。
- 不把所有主题一次性设为默认可见选项，优先保证质量和可维护性。

## 推荐上线范围

第一阶段建议接入低风险主题：

| 分组 | 主题 |
|---|---|
| 基础 | `auto`、`light`、`dark` |
| 专注 | `forest`、`midnight` |
| 柔和 | `monet`、`morandi`、`sakura-mist` |
| Stitch | `healing-sea-blue`、`mint-tea-green`、`klein-gold` |

第二阶段再接入艺术和趣味主题：

| 类型 | 说明 |
|---|---|
| 艺术主题 | `starry-night`、`qingming-scroll`、`cezanne`、`sisley`、`pissarro`、`gauguin`、`matisse`、`qi-baishi` |
| 高饱和主题 | `rainbow-rgb`、`neon-track`、`eva`、`p-site`、`orange-slate` |
| 强风格主题 | `dongbei-rain`、`bleach-tybw`、`cream-berry-purple`、`seafoam-apricot`、`honey-sunset`、`crimson-ivory` |

高饱和主题需要单独验收按钮文本、焦点环、状态提示和表格 hover 的对比度。

## Token 映射方案

SafeVault 核心 token 到当前项目 token 的建议映射如下：

| SafeVault token | 当前项目 token | 处理方式 |
|---|---|---|
| `Accent` | `--accent` | 作为主交互色、链接色、主要按钮色 |
| `Accent Strong` | `--accent-2` | 作为强按钮、active 渐变第二色 |
| `Signal` | `--success` | 作为成功状态、正向指标色 |
| `Danger` | `--danger` | 作为错误、危险操作、校验失败色 |
| `BG` | `--bg` | 页面主背景 |
| `BG Deep` | `--surface`、`--sidebar`、部分暗层 | 按主题明暗决定是否用于侧栏和深层容器 |
| `Ink` | `--text` | 主文本 |
| `Muted` | `--muted`、`--nav-text`、`--tab-text` | 次级文本和导航文本 |
| `Surface` | `--panel`、`--field-bg`、`--chip-bg` | 卡片、表单、菜单底色 |
| `Surface Dark` | `--panel-2`、`--field-bg-soft`、`--table-alt` | 次级面板、表格交替行、弱背景 |

当前项目缺少的派生 token 需要由核心色生成：

- `--line`、`--card-border`、`--field-border`：由 `Muted` 或 `Accent` 低透明度混合生成。
- `--accent-soft-*`：由 `Accent` 和 `Surface` 混合生成。
- `--danger-bg`、`--danger-line`：由 `Danger` 和 `Surface` 混合生成。
- `--status-*`：按 info / success / error 三类派生，不直接复制主色。
- `--control-active-text`：不能固定为白色，需要按主题深浅设置，避免 `klein-gold`、`neon-track` 等主题失去可读性。
- `--focus-ring`：使用 `Accent` 半透明派生，保证键盘焦点清晰但不刺眼。
- `--shadow`、`--card-shadow-*`：按明暗主题区分，浅色用低透明深色阴影，暗色用黑色阴影。

## 具体改动点

### 1. 主题目录和元数据

在 `web.py` 的 JS 中把当前 `themeModes` 扩展为主题目录：

```js
const themeCatalog = [
  { id: 'auto', label: '跟随系统', group: '基础主题', system: true },
  { id: 'light', label: '默认浅色', group: '基础主题', scheme: 'light' },
  { id: 'dark', label: '默认深色', group: '基础主题', scheme: 'dark' },
  { id: 'forest', label: '森林安全', group: '专注主题', scheme: 'light' }
];
```

`system` 不建议作为新 ID 引入，避免和当前 `auto` 存储值冲突。SafeVault 的 `system` 映射为当前项目的 `auto`。

### 2. 设置菜单加入外观区域

当前项目已经有设置弹窗，可以增加一个“外观”区域：

- 主题选择下拉框或分组列表。
- 每个主题显示 3 个色块：主色、背景、文字。
- 当前主题显示在顶部主题按钮上。
- 顶部主题按钮不再循环 30 个主题；推荐点击后打开设置中的外观区域，或者只在 `auto/light/dark` 之间快速切换。

### 3. CSS 主题块

新增主题 CSS 块：

```css
:root[data-theme="forest"] {
  color-scheme: light;
  --accent: #2f6b3f;
  --accent-2: #1f4d2b;
  --success: #0f766e;
  --danger: #b54432;
  --bg: #f3f7f0;
  --text: #172313;
}
```

实际落地时不能只写这几个变量。每个主题至少要覆盖：

- 基础：`--bg`、`--surface`、`--panel`、`--panel-2`、`--text`、`--muted`、`--line`
- 品牌与状态：`--accent`、`--accent-2`、`--success`、`--warning`、`--danger`
- 交互：`--accent-soft-*`、`--control-*`、`--focus-ring`
- 表单与卡片：`--field-*`、`--card-*`
- 状态提示：`--status-info-*`、`--status-success-*`、`--status-error-*`
- 数据区域：`--table-alt`、`--table-hover`、滚动条 token

### 4. 清理固定色

需要把以下固定色改成 token 或派生变量：

- 按钮和链接里的 `#fff`。
- metric 第 5 至第 7 项固定色。
- loading/progress 中固定蓝绿橙渐变。
- `results` 滚动条固定色。
- 少量浅蓝提示背景，例如固定 `#eff6ff`、`#dbe7f4`。
- 暗色专属选择器里只覆盖 `.pack-name-*`，后续设置弹窗和新主题菜单也要纳入同一规则。

### 5. 主题解析逻辑

当前逻辑：

```js
auto -> 系统 dark/light
dark -> dark
light -> light
```

建议改为：

```js
auto -> 系统 dark/light
light/dark -> 自身
其他主题 -> 自身，同时根据 themeCatalog.scheme 设置 colorScheme
```

这样 `document.documentElement.dataset.theme` 可以是 `forest`、`midnight` 等真实主题 ID，CSS 用 `:root[data-theme="forest"]` 命中。

### 6. 测试和验收

需要补充 `tests/test_web_ui_contract.py`：

- 断言主题目录存在。
- 断言设置菜单出现主题选择。
- 断言关键主题 ID 出现在 HTML。
- 断言 `auto` 仍然存在，避免破坏已有用户设置。
- 断言 `applyThemeMode` 支持非 `dark/light` 的主题 ID。

手动或浏览器验收：

- `auto` 跟随系统明暗。
- `light/dark` 行为不变。
- 至少验收 `forest`、`midnight`、`monet`、`sakura-mist`、`klein-gold`。
- 检查 375px、768px、1440px 宽度。
- 检查设置弹窗、语言下拉框、模型下拉框、JAR 树状筛选、结果表格、API 日志、loading 状态。
- 检查键盘 focus ring 可见。
- 检查错误、成功、警告不能只依赖颜色表达。

## 实施阶段

### 阶段 1：建立主题基础设施

- 增加 `themeCatalog`。
- 修改 `resolveThemeMode` / `applyThemeMode`。
- 设置菜单增加主题选择。
- 保持 `auto/light/dark` 兼容。
- 补 UI 契约测试。

交付标准：功能不改变视觉主题数量，但主题选择架构可以承载更多主题。

### 阶段 2：迁移低风险主题

- 接入 `forest`、`midnight`、`monet`、`morandi`、`sakura-mist`。
- 建立 SafeVault 核心 token 到当前派生 token 的固定映射。
- 清理明显固定色。
- 做基础对比度和页面截屏检查。

交付标准：用户能在设置中选择新主题，主要工作流无可读性问题。

### 阶段 3：接入完整主题库

- 接入剩余 SafeVault Web 主题。
- 趣味、高饱和、联名主题放入独立分组。
- 对 `rainbow-rgb` 这类特殊主题单独保留渐变或动画 token，但要尊重 `prefers-reduced-motion`。

交付标准：主题列表完整，所有主题下核心控件可读。

### 阶段 4：维护性收敛

如果主题数量继续增加，建议把主题定义从 `INDEX_HTML` 中抽出：

- `src/mc_mod_i18n/theme_tokens.py`：维护主题 token 数据。
- `render_theme_css()`：生成 CSS。
- `render_theme_catalog_json()`：生成前端主题目录。

当前阶段可以先不做这一步，避免扩大改动面。

## 风险

- SafeVault 核心 token 数量少于当前 UI 派生 token，直接映射会导致下拉框、表格、状态提示细节变脏。
- 高饱和主题容易让按钮文字、焦点环和状态色失去对比度。
- `Surface` 是半透明色时，当前页面多层容器叠加可能导致视觉不稳定，需要必要时转成不透明底色。
- 主题数量过多时，顶部按钮循环体验差，必须放进设置菜单或主题选择器。
- 当前 UI CSS 在 Python 字符串里，主题块过多会让 `web.py` 继续变长，后续需要考虑抽离。

## 推荐决策

建议采用“先基础设施，后主题库”的路线。

第一版只做主题选择能力和 5 到 8 个低风险主题，确认当前工作台的表单、表格、下拉框、loading、报告视图都稳定后，再把 SafeVault 剩余主题完整接入。这样能避免一次性引入大量色值后出现难定位的可读性问题。
