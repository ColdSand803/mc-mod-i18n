# mc-mod-i18n 实用补强方案

> 目标：围绕 Minecraft Mod 汉化的真实工作流，把当前 CLI + Web UI 从“能跑”补到“好用、可复用、可批量处理”。

## 1. 当前项目的主要问题

> 已有能力基线：validator.py 已做 printf/brace/颜色代码校验并在 CLI 流程中调用；OpenAICompatibleTranslator 已实现指数退避重试+429 协调+JSONL debug 日志；CLI 已支持并发处理多个 JAR 和 nested JAR 递归扫描。以下问题聚焦在”有但不全”和”确实缺”两个层面。

1. 功能偏 MVP，核心链路能跑通，但离日常批量使用还有明显距离。
2. **校验覆盖不够**：已有占位符/颜色代码校验，但缺翻译前预检（空键、重复键、超长文本），校验失败时只能回退原文，缺”标红+一键重译”的交互闭环。
3. **术语表和翻译记忆没有持久化复用**：GlossaryTranslator 的内置词表和外部 glossary.json 只在单次运行生效，翻译结果不回流积累，重复劳动多。
4. **缺批量任务队列和增量更新**：CLI 支持并发处理多个 JAR，但 Web UI 侧没有任务队列视图；没有基于 hash 的增量判断，每次都要全量重跑。
5. Web UI 是单文件 inlined HTML，功能更像操作面板，缺少任务进度管理、错误集中定位、历史结果回看。
6. 重试和日志已有基础，但**断点续跑**（中断后恢复，不必整批重来）和**按任务归档日志**还没做，排查问题时上下文分散。

## 2. 优先补充的实用功能

### P0：先补稳定性和可用性（加固已有+补齐缺口）

- 翻译前预检（新增）
  - 扫描语言文件数量、空键、重复键、超长文本、特殊占位符。
  - 输出明确的风险列表，避免翻译完才发现问题。
- 翻译结果校验（增强）
  - 现有 validator 已检查 `%s`、`%1$s`、`{0}`、`§a`。补充 `\n` 换行一致性、键值对数量完整性检查。
  - 校验失败的条目在报告中单独标红，UI 侧支持一键回退或指定 provider 重新翻译。
- 失败重试与断点续跑（增强+新增）
  - 现有 OpenAICompatibleTranslator 已有指数退避+429 协调，保持不变。
  - 新增任务级断点续跑：记录每个 JAR/文件的处理状态，中断后只处理未完成的部分。
- 更完整的日志与错误定位（增强）
  - 现有 JSONL debug 日志保留，在其基础上增加按 task_id 分组。
  - 日志记录 provider、模型、术语表版本（hash）、输入文件 hash、每条失败的原因摘要。
  - 报告页增加错误聚合视图：按失败类型、JAR、provider 分组展示。

### P1：提高日常效率

- 术语库管理
  - 支持多个 glossary 并存，可启用/禁用单个词表。
  - 支持项目级（`.mc-i18n/glossary.json`）和全局级（`~/.mc-i18n/glossary.json`）自动发现合并。
  - 支持导入导出为标准 JSON/CSV，便于跨项目积累。
- 翻译记忆（TM）
  - 同一英文 key 或相似句式（编辑距离阈值匹配）优先复用历史译文。
  - TM 数据本地持久化，按 source→target 映射存储，降低重复调用 AI 的成本。
- 批量任务队列
  - Web UI 侧增加任务队列视图：一次上传多个 Mod/JAR。
  - 显示每个任务的进度、状态、耗时、失败条目数。
  - 已完成任务保留在列表中，可回看和导出。
- 增量更新
  - 基于文件 hash（SHA-256）和 key hash 判断是否需要重新翻译。
  - 只翻译新增或变更的条目，未变条目直接复用上一次结果。
- 更强的筛选和搜索
  - 按文件、命名空间、文本长度、风险类型、翻译状态筛选。
  - 支持按关键字快速定位条目，搜索结果高亮。

### P1：UI 增强（适配当前单文件架构）

> 当前 Web UI 是 web.py 内嵌的单文件 HTML（~2000 行 CSS + JS），CSS 变量体系已建立，sidebar 布局已就位。以下优化在这个基础上做，不引入前端构建工具链。

- 任务面板视图
  - 把当前单次翻译流程重构为”任务卡片”模型：每个上传/翻译产生一张卡片，独立显示进度和状态。
  - 任务完成后卡片保留，可展开查看详情（条目数、失败数、耗时）。
- 错误集中定位
  - 报告区的错误条目不再散落在长列表中，改为”错误聚焦视图”：一键只看失败项，按错误类型分组。
- 信息层级统一
  - 统一按钮、标签、状态、空状态的视觉样式，利用已有 CSS 变量（`--danger`/`--success`/`--warning`/`--muted`）。
  - 翻译结果、风险提示、任务状态在 UI 上一眼可分。
- 响应式加固
  - 当前 grid 布局在窄屏下 sidebar 可能挤压内容区。增加 `max-width` 断点下的 sidebar 折叠逻辑。
  - 长文本单元格加 `word-break` 和 `overflow-wrap`，避免撑破表格。
- 动效和性能
  - 保留现有 CSS transition（`var(--motion-fast)`/`var(--motion-base)`），不新增动画。
  - 大列表渲染改为虚拟滚动或分页，避免 DOM 节点过多导致卡顿。
  - 对动效提供 `prefers-reduced-motion` 适配。

### P2：增强产品完成度

- 结果对比视图
  - 原文、机器译文、人工修订并排展示。
  - 方便审校和微调。
- 导出策略配置
  - 支持只导出完成项、只导出修订项、或全量导出。
  - 支持自定义输出目录和命名规则。
- 任务历史
  - 记录每次翻译的输入、参数、产物。
  - 可以回看旧版本，支持回滚。
- 配置预设
  - 把常用 provider、模型、API key 环境变量、术语表、校验规则保存为 preset。

## 3. 建议的实施顺序

### 第一阶段：先让结果可信

1. 做翻译前预检（新增）。
2. 增强翻译结果校验，补换行一致性、键数量完整性。
3. 补任务级日志分组和错误聚合视图。
4. 做断点续跑。
5. **验收关卡**：对 3 个不同规模的 Mod JAR（含 1 个 nested JAR）跑翻译，确认预检→翻译→校验→报告链路完整，失败条目能精确定位到 key。

### 第二阶段：让流程省事

1. 做术语库管理（多 glossary + 自动发现）。
2. 做翻译记忆持久化。
3. Web UI 侧增加任务队列视图和错误聚焦视图。
4. 做增量更新（hash 判断）。
5. 增强搜索、筛选，做 UI 响应式加固和大列表性能优化。
6. **验收关卡**：同一批 JAR 跑两遍，第二遍全部命中增量跳过；术语库和 TM 数据在第二次运行时被复用。

### 第三阶段：让工具可长期使用

1. 做结果对比视图和任务历史。
2. 做导出策略与配置预设。
3. 补回归测试：覆盖常见 Mod 结构、nested JAR、新旧 lang 格式、主流 AI provider 的响应格式兼容。
4. 检查 PyInstaller 打包体积和新依赖的 Windows 兼容性。
5. **验收关卡**：`python -m pytest` 全绿；`.\scripts\build_exe.ps1 -Clean` 打包成功且 exe 启动 UI 正常。

## 4. 需要重点盯住的问题

- nested JAR 扫描是否完整，是否会漏掉 JarJar / bundled 模式。
- 占位符和颜色代码是否在不同 provider 下稳定保留（已有 validator 打底，需持续关注误杀/漏杀）。
- 大型 Mod 包处理时的内存、耗时和 UI 响应。
- API provider 兼容性和限流失败的处理（已有 RateLimiter 协调并发 429，新增 provider 时需验证）。
- 输出资源包结构是否适配常见 Minecraft 版本（pack_format 版本映射）。
- 硬编码映射工作台和普通翻译流程是否能统一存档与复用。
- UI 在长列表、大文本、窄窗口下是否会溢出、截断错位或遮挡操作按钮。
- 新功能引入的依赖是否增加 PyInstaller 打包体积，是否在 Windows 下兼容（尤其纯 Python 依赖优先，避免 C 扩展）。
- 翻译记忆和术语库的本地存储格式需考虑跨版本兼容，避免格式变更导致用户积累的数据失效。

## 5. 推荐落地方式

做一个”质量优先版”（基于已有 validator/retry/logging 加固，而非重写）：

1. **预检**（新增）：扫描→风险列表→阻断或警告。
2. **校验增强**（扩展现有 validator）：补换行一致性、键数量检查，校验失败在报告中标红。
3. **日志分组**（扩展现有 JSONL debug log）：按 task_id 分组，补错误聚合视图。
4. **断点续跑**（新增）：记录处理进度，中断后可恢复。

这四项直接改善可用性，也为后面的术语库、翻译记忆、批量队列打基础。  
如果只做界面优化，不先补这些底层能力，后面还是会反复返工。

实施时注意：改动集中在 `validator.py`、`translator.py`、`cli.py`、`report.py`，不重写架构，只在现有接口上扩展参数和返回值。
