# Minecraft Mod JAR 全自动汉化工具方案

## 1. 目标

实现一个可以导入 Minecraft Mod JAR 包并自动生成中文语言资源的工具。第一阶段优先保证稳定处理标准语言文件，不直接修改原 JAR，默认输出资源包 ZIP。

核心流程：

```text
导入 JAR / mods 文件夹
  -> 识别 Mod 信息
  -> 扫描 assets/**/lang/en_us.json 或 en_us.lang
  -> 提取待翻译文本
  -> 调用翻译器
  -> 校验占位符、颜色代码和格式
  -> 生成 zh_cn.json / zh_cn.lang
  -> 打包为资源包 ZIP
  -> 输出报告
```

## 2. 第一阶段 MVP

第一阶段只处理资源文件汉化，避免修改 JAR 带来的签名失效和兼容性风险。

必须完成：

- CLI 命令：`mc-mod-i18n translate <jar-or-folder>`
- 支持单个 JAR 和 mods 文件夹批量处理
- 自动识别 Fabric / Quilt / Forge / NeoForge 基础元数据
- 解析 `assets/<modid>/lang/en_us.json`
- 解析旧版 `assets/<modid>/lang/en_us.lang`
- 如果 JAR 内已有 `zh_cn`，默认复用已有翻译，只补全缺失项
- 支持术语表 `glossary.json`
- 校验 `%s`、`%1$s`、`{0}`、`§a` 等占位符和格式符
- 输出资源包 ZIP
- 生成 HTML 报告

第一阶段翻译器设计：

- `copy`：复制原文，方便测试流程
- `glossary`：基于内置和用户术语表做规则替换
- `openai-compatible`：预留 OpenAI 兼容 Chat Completions 接口，用于真实自动翻译

## 3. 第二阶段增强

- 图形界面：拖拽 JAR、预览、校对、导出
- 翻译缓存：SQLite，按 `source + key` 复用历史翻译
- 断点续翻：失败后可继续
- 多语言目标：`zh_tw`、`ja_jp` 等
- 术语表 UI：添加、编辑、导入、导出
- 对比报告：英文、已有中文、新生成中文
- 支持直接输出修改版 JAR，但默认关闭

## 4. 第三阶段高级能力

- 扫描 `.class` 中疑似硬编码文本，只生成报告，不默认改字节码
- 接入 Modrinth / CurseForge 元数据，辅助识别 Mod 名称和版本
- 从社区汉化仓库导入已有翻译
- 按整合包批量生成一个总资源包
- 对翻译质量做二次 AI 审校

## 5. 技术选型

当前仓库先使用 Python 标准库实现 CLI MVP：

- 不依赖外部包，便于直接运行
- 使用 `zipfile` 读取 JAR 和生成资源包
- 使用 `argparse` 实现命令行
- 使用 `json` 解析现代语言文件
- 使用简单 parser 解析旧版 `.lang`
- 使用 `urllib.request` 实现 OpenAI 兼容 HTTP 翻译器

后续如果需要桌面应用，再在 CLI 核心外包一层 Tauri / Electron / JavaFX。

## 7. CLI 设计

```bash
python -m mc_mod_i18n translate ./mods/example.jar --out ./dist
python -m mc_mod_i18n translate ./mods --provider glossary --glossary glossary.json
python -m mc_mod_i18n translate ./mods --provider openai-compatible --model gpt-4o-mini
```

常用参数：

- `--source-locale en_us`
- `--target-locale zh_cn`
- `--out dist`
- `--provider copy|glossary|openai-compatible`
- `--glossary glossary.json`
- `--overwrite-existing`
- `--pack-format 15`
- `--api-url https://api.openai.com/v1/chat/completions`
- `--api-key-env OPENAI_API_KEY`
- `--model <model>`

## 9. 校验规则

翻译后必须校验：

- 原文中的 `%s`、`%1$s`、`%d` 等 printf 占位符不能丢
- `{0}`、`{name}` 等花括号占位符不能丢
- `§a`、`§r` 等 Minecraft 颜色/格式代码不能丢
- JSON 必须能重新序列化
- 翻译不能为空

校验失败时：

- 保留原文或已有中文
- 在报告中标记失败原因
- 不阻塞整个资源包生成
