# 添加体系语言切换功能方案

## 背景

当前 Web UI 的界面文案、状态提示、错误提示、报告标题基本以中文硬编码在 `src/mc_mod_i18n/web.py` 的 `INDEX_HTML` 和 Python 服务端逻辑中。工具已经有“源语言”和“目标语言”下拉框，但这两个字段服务于 Minecraft 语言文件翻译，分别对应 `source_locale` 和 `target_locale`，不是工具自身界面语言。

因此需要新增一套“体系语言”能力，用于切换工具自身 UI、提示、报告和部分错误信息的显示语言。该能力必须与 Minecraft 内容翻译语言彻底隔离，避免用户切换界面语言时误改翻译任务的源语言或目标语言。

## 术语定义

- **体系语言 / UI 语言**：工具自身界面、按钮、提示、报告、错误信息使用的语言。建议字段名使用 `ui_locale`。
- **源语言**：待翻译 Minecraft 语言文件或 FTB Quests 文件的原始语言。现有字段为 `source_locale`。
- **目标语言**：翻译输出语言。现有字段为 `target_locale`。
- **Minecraft locale**：例如 `en_us`、`zh_cn`、`ja_jp`，用于内容翻译。
- **UI locale**：建议也使用 Minecraft 风格小写下划线，例如 `zh_cn`、`en_us`、`zh_tw`，但语义上只表示工具界面语言。
- **扩展语言包**：用户额外提供的工具界面语言 JSON，例如 `ja_jp.json`、`fr_fr.json`，用于补充内置界面语言。
- **直接 JSON 翻译模式**：不上传 JAR，而是直接上传语言 JSON，解析 key-value 后翻译 value，输出目标 locale JSON。

## 目标

1. Web UI 支持切换工具界面语言，第一阶段至少支持 `zh_cn` 和 `en_us`。
2. 体系语言切换不影响 `source_locale`、`target_locale`、翻译缓存、翻译结果和输出内容。
3. 用户选择的体系语言可持久化，刷新页面后保持。
4. 动态状态文案、加载进度、结果摘要、下载按钮、空状态、表单帮助文本等主要 UI 文案跟随体系语言切换。
5. 服务端返回的用户可见错误信息、HTML 报告、JSON 报告中的标题或状态说明逐步支持体系语言。
6. 所有未翻译文案都有明确 fallback，避免界面出现空白。
7. 支持导出当前项目的英文基准语言 JSON，方便小语种用户自行翻译。
8. 支持导入扩展语言包，导入后在界面语言下拉框中可选。
9. 新增直接 JSON 翻译模式，用户可上传 `en_us.json` 并用本工具翻译为 `ja_jp.json`、`fr_fr.json` 等目标语言文件。

## 非目标

1. 不自动翻译 Mod 内容，不改变源语言和目标语言选择逻辑。
2. 不在第一阶段引入复杂前端构建工具链，继续适配当前单文件 HTML 架构。
3. 不要求一次性覆盖所有历史文档和开发者注释。
4. 不把 `ui_locale` 写入翻译配置 hash，避免同一翻译任务因界面语言不同而重复调用 API。
5. 不做在线语言包下载，语言包随程序发布。
6. 不把任意 JSON 都当成可翻译配置文件处理，第一阶段只支持“扁平 key-value 文本 JSON”和本项目 UI 语言包 schema。
7. 不在第一阶段做语言包市场、在线同步、社区审核或自动更新。

## 当前状态

### 已有能力

- `source_locale` 默认 `en_us`，`target_locale` 默认 `zh_cn`。
- 语言下拉框已经有 Minecraft locale 列表、搜索、自定义 locale 输入。
- Web UI 主题选择、设置弹窗、任务进度、结果页都在单个 `INDEX_HTML` 内实现。
- 服务端 API 以 JSON 返回任务状态和结果，进度阶段已有结构化字段。

### 主要缺口

- UI 静态文本硬编码在 HTML 中。
- JS 动态拼接文本硬编码在函数中，例如状态栏、进度提示、结果摘要、弹窗。
- Python 服务端错误信息直接抛中文字符串。
- 报告生成函数使用固定中文标题和字段名。
- 缺少“体系语言”和“内容语言”的命名边界。
- 缺少直接上传 `.json` 语言文件并输出目标语言 JSON 的通用能力。
- 缺少扩展语言包的导入、校验、存储和回退策略。

## 设计原则

1. **命名隔离**：体系语言统一叫 `ui_locale`，绝不复用 `source_locale` 或 `target_locale`。
2. **渐进替换**：先覆盖用户高频可见文案，再逐步清理低频提示和报告。
3. **结构化 key**：文案用稳定 key 管理，不在 JS/Python 中继续散落字符串。
4. **fallback 明确**：查不到当前语言文案时回退到 `zh_cn`，再回退到 key 本身。
5. **缓存无关**：翻译缓存只与输入、翻译配置、术语表等内容相关，和 UI 语言无关。
6. **状态代码化**：服务端尽量返回状态 code，前端按 `ui_locale` 渲染成文本。

## 语言支持范围

### 第一阶段

- `zh_cn`：简体中文，默认语言。
- `en_us`：English (US)。

### 第二阶段可选

- `zh_tw`：繁体中文。
- `ja_jp`：日本語。
- `ko_kr`：한국어。

语言越多，维护成本越高。建议第一阶段只做 `zh_cn` 和 `en_us`，把机制做稳后再扩展。

## 交互方案

### 入口位置

在顶部 header 右侧增加体系语言切换控件，放在主题选择和设置按钮附近。

推荐形态：

- 图标：`ri-translate-2` 或 `ri-global-line`。
- 显示值：`简体中文` / `English`。
- 下拉选项：显示语言名和 locale code，例如 `简体中文 zh_cn`、`English en_us`。

### 初始语言决策

页面加载时按以下顺序决定 `ui_locale`：

1. URL 参数：`?ui_locale=en_us`。
2. `localStorage`：`mc-mod-i18n-ui-locale`。
3. 浏览器语言：`navigator.language`，例如 `zh-CN` 映射到 `zh_cn`。
4. 默认值：`zh_cn`。

### 持久化行为

- 用户手动切换后写入 `localStorage`。
- 表单提交时通过隐藏字段 `name="ui_locale"` 提交给服务端。
- 刷新页面后保持上次选择。
- 清空缓存不影响体系语言设置。

### 与源语言自动识别的关系

FTB Quests 上传 `zh_cn.snbt` 后，源语言下拉框自动切到 `zh_cn`，这是内容语言识别。

体系语言切换不会参与该逻辑：

- 切 UI 语言到 English，不改变 `source_locale`。
- 上传 `zh_cn.snbt`，只改变 `source_locale`，不改变 `ui_locale`。
- 生成 `zh_cn` 输出，取决于 `target_locale`，不取决于 `ui_locale`。

### 扩展语言包工作流

小语种用户的完整路径：

1. 在设置中点击“下载英文界面语言包”。
2. 得到 `mc-mod-i18n-ui-en_us.json`。
3. 切换输入类型为“语言 JSON 文件”。
4. 上传该 JSON。
5. 源语言选择 `en_us`，目标语言选择目标 locale，例如 `fr_fr`。
6. 使用当前翻译器生成 `mc-mod-i18n-ui-fr_fr.json`。
7. 回到设置，导入生成的 `fr_fr` 扩展语言包。
8. 在界面语言下拉框中选择 `Français`。

普通 Minecraft lang JSON 用户的路径：

1. 切换输入类型为“语言 JSON 文件”。
2. 上传普通 `en_us.json`。
3. 选择目标语言。
4. 输出同 key 的目标 JSON。
5. 用户自行放回资源包或 Mod 项目。

## 技术方案

### 前端文案资源

当前不引入构建工具，建议先在 `INDEX_HTML` 的 `<script>` 内维护轻量字典：

```js
const uiLocales = [
  { code: 'zh_cn', label: '简体中文', nativeLabel: '简体中文' },
  { code: 'en_us', label: 'English', nativeLabel: 'English' }
];

const uiMessages = {
  zh_cn: {
    'app.title': '生成汉化资源包',
    'nav.workspace': '工作区',
    'form.source_locale': '源语言',
    'status.waiting_jar': '等待选择 JAR。'
  },
  en_us: {
    'app.title': 'Generate Localization Pack',
    'nav.workspace': 'Workspace',
    'form.source_locale': 'Source Language',
    'status.waiting_jar': 'Waiting for JAR files.'
  }
};
```

新增工具函数：

```js
function t(key, params = {}) {
  const catalog = uiMessages[currentUiLocale] || uiMessages.zh_cn;
  const fallback = uiMessages.zh_cn || {};
  const template = catalog[key] || fallback[key] || key;
  return formatMessage(template, params);
}
```

支持简单占位符：

```text
完成：处理 {count} 个 JAR。
Done: processed {count} JAR files.
```

### HTML 标记

静态节点使用 `data-i18n` 标记：

```html
<h1 data-i18n="app.title">生成汉化资源包</h1>
<span data-i18n="nav.workspace">工作区</span>
<input data-i18n-placeholder="locale.search" placeholder="搜索代码或语言">
```

切换语言时调用：

```js
function applyUiLocale(locale) {
  currentUiLocale = resolveUiLocale(locale);
  document.documentElement.lang = uiLocaleToHtmlLang(currentUiLocale);
  document.querySelectorAll('[data-i18n]').forEach(node => {
    node.textContent = t(node.dataset.i18n);
  });
  document.querySelectorAll('[data-i18n-placeholder]').forEach(node => {
    node.placeholder = t(node.dataset.i18nPlaceholder);
  });
}
```

### JS 动态文本

所有动态文本从直接拼接改为 `t()`。

重点替换区域：

- `syncInputKind()` 的等待状态。
- `syncFiles()` 的文件选择提示。
- `renderLoading()` 的进度文案。
- 任务完成后的 `statusBox.textContent`。
- 下载文件命名弹窗。
- 错误/空状态/搜索无结果提示。
- 结果页 metric label。
- 硬编码报告、API 日志、语言工作区表头。

示例：

```js
statusBox.textContent = isFtbquests
  ? t('status.waiting_ftbquests')
  : t('status.waiting_jar');
```

### 服务端文案资源

建议新增模块：

```text
src/mc_mod_i18n/ui_i18n.py
src/mc_mod_i18n/locales/zh_cn.json
src/mc_mod_i18n/locales/en_us.json
```

`ui_i18n.py` 提供：

- `normalize_ui_locale(value: str) -> str`
- `resolve_ui_locale(value: str | None) -> str`
- `translate_ui(key: str, locale: str, **params) -> str`
- `load_ui_catalog(locale: str) -> dict[str, str]`

服务端 API 中读取：

```python
ui_locale = resolve_ui_locale(fields.get("ui_locale"))
```

服务端不再直接抛用户可见中文字符串，优先抛带 key 的错误：

```python
raise UserFacingError("error.ftbquests_missing_input")
```

统一返回时再转成本地化消息：

```python
payload = {"ok": False, "error": translate_ui(error.key, ui_locale)}
```

### 报告本地化

报告生成函数新增可选参数：

```python
write_report(path, entries, ..., ui_locale="zh_cn")
write_ftbquests_html_report(path, result, ui_locale="zh_cn")
write_hardcoded_report(path, entries, ui_locale="zh_cn")
```

报告本地化范围：

- HTML `<title>`。
- 页面标题。
- 表头。
- 状态标签。
- 汇总指标名称。
- 错误原因的用户可见描述。

JSON 报告建议保留机器可读字段名不变，只增加可选显示字段：

```json
{
  "status": "ok",
  "status_label": "成功"
}
```

这样不会破坏现有脚本消费 JSON 的兼容性。

### 扩展语言包机制

扩展语言包建议采用“带元数据的 JSON schema”，同时兼容扁平 key-value JSON。

推荐格式：

```json
{
  "schema_version": 1,
  "locale": "fr_fr",
  "name": "French",
  "native_name": "Français",
  "source_locale": "en_us",
  "source_version": "0.1.0",
  "messages": {
    "app.title": "Generate Localization Pack",
    "nav.workspace": "Workspace",
    "status.done": "Done: processed {count} files."
  }
}
```

兼容格式：

```json
{
  "app.title": "Generate Localization Pack",
  "nav.workspace": "Workspace",
  "status.done": "Done: processed {count} files."
}
```

兼容格式导入时由系统补齐元数据：

- `schema_version`: `1`
- `locale`: 根据文件名推断，例如 `fr_fr.json`
- `name`: 根据 locale 或文件名生成
- `source_locale`: 默认 `en_us`
- `messages`: 原 JSON 根对象

语言包存储位置：

```text
<cache_dir>/ui-locales/
  fr_fr.json
  ja_jp.json
  zh_tw.json
```

加载顺序：

1. 内置 `zh_cn`、`en_us`。
2. 用户扩展语言包目录。
3. 同 locale 的扩展语言包覆盖内置非默认语言，但不能覆盖 `zh_cn` 和 `en_us` 的内置基准。

校验规则：

- `locale` 必须是合法 locale：`^[a-z0-9_]{2,24}$`。
- `messages` 必须是对象。
- key 必须是非空字符串。
- value 第一阶段只接受字符串。
- 占位符必须与 `en_us` 基准一致，例如 `{count}`、`{filename}`。
- HTML 标签、换行标记、Minecraft 格式码等受保护片段不能丢失。
- 缺失 key 允许导入，但 UI 渲染时必须 fallback。

### 英文基准语言包导出

为小语种用户提供“下载英文语言包”入口。

导出内容：

- `en_us.json`
- 包含完整 key。
- 包含 schema 元数据。
- 包含当前项目版本或语言包版本。

接口建议：

```text
GET /api/ui-locales/export/en_us
```

返回文件名：

```text
mc-mod-i18n-ui-en_us.json
```

导出的英文基准语言包可以作为直接 JSON 翻译模式的输入，也可以交给人工或其他 CAT 工具翻译。

### 扩展语言包导入

新增“导入界面语言包”入口，建议放在设置弹窗中，不放在主翻译表单里，避免和 Mod 翻译输入混淆。

导入流程：

1. 用户上传 `fr_fr.json`。
2. 前端提交到 `/api/ui-locales/import`。
3. 服务端解析 JSON。
4. 推断或读取 `locale`。
5. 校验 schema、key、占位符。
6. 写入 `<cache_dir>/ui-locales/fr_fr.json`。
7. 返回导入结果，包括缺失 key 数、占位符风险数、可用 locale。
8. 前端刷新界面语言下拉框。

导入后如果当前选择该语言：

- 立即调用 `applyUiLocale(locale)`。
- 缺失文案按 fallback 显示。
- 设置状态里提示“已导入，缺失 N 个 key，将回退显示”。

### 直接 JSON 翻译模式

新增输入类型：

```text
语言 JSON 文件
```

该模式不是 JAR 翻译，也不是 FTB Quests 翻译。它处理单个或多个 JSON 语言文件，用于：

- 翻译本项目 UI 语言包。
- 翻译普通 Minecraft `assets/<modid>/lang/en_us.json`。
- 翻译其他扁平 key-value 语言 JSON。

支持输入：

```text
en_us.json
mc-mod-i18n-ui-en_us.json
assets/example/lang/en_us.json
```

处理规则：

- 只翻译字符串 value。
- 保留所有 key。
- 保留 JSON 对象结构。
- 默认按扁平 key-value 处理。
- 如果检测到 `{ "messages": { ... } }`，只翻译 `messages` 内部 value。
- 非字符串 value 第一阶段跳过并报告。
- 空字符串跳过。
- 已存在目标 JSON 时可选择“只补缺失 key”或“覆盖全部 value”。

输出：

```text
fr_fr.json
mc-mod-i18n-ui-fr_fr.json
```

如果输入是 UI 语言包 schema，输出也保留 schema：

```json
{
  "schema_version": 1,
  "locale": "fr_fr",
  "name": "French",
  "native_name": "Français",
  "source_locale": "en_us",
  "source_version": "0.1.0",
  "messages": {
    "app.title": "... translated ..."
  }
}
```

如果输入是普通 Minecraft lang JSON，输出为扁平 JSON：

```json
{
  "block.example.name": "... translated ...",
  "item.example.name": "... translated ..."
}
```

### JSON 翻译校验

直接 JSON 翻译模式需要比普通 Mod 翻译多做 key 完整性校验。

校验项：

- 输出 key 集合必须与输入 key 集合一致，除非用户选择“只输出变更项”。
- 所有占位符必须保留。
- `{count}`、`{name}` 这类 ICU 风格占位符不能被翻译。
- `%s`、`%1$s`、`{0}` 必须保留。
- HTML 标签和实体不能破坏。
- `\n`、`\t` 等转义字符必须保留语义。
- JSON 必须可解析。
- 输出必须使用 UTF-8。

报告内容：

- 总 key 数。
- 已翻译 value 数。
- 跳过非字符串 value 数。
- 占位符风险数。
- 缺失或多余 key 数。
- 可直接导入为 UI 语言包的判断结果。

## 数据结构

### 隐藏字段

在表单中新增：

```html
<input type="hidden" name="ui_locale" id="ui_locale" value="zh_cn">
```

### 本地存储

```text
mc-mod-i18n-ui-locale = zh_cn | en_us | zh_tw ...
```

扩展语言包存储在服务端缓存目录，不放入浏览器 localStorage：

```text
<cache_dir>/ui-locales/<locale>.json
```

### API 字段

所有需要用户可见文案的 API 可接受：

```json
{
  "ui_locale": "en_us"
}
```

翻译任务请求仍保留：

```text
source_locale=en_us
target_locale=zh_cn
```

直接 JSON 翻译请求新增：

```text
input_kind=json
json_files=<uploaded files>
source_locale=en_us
target_locale=fr_fr
json_output_mode=full | missing_only | changed_only
json_schema=auto | flat | ui_locale
```

扩展语言包 API：

```text
GET  /api/ui-locales
GET  /api/ui-locales/export/en_us
POST /api/ui-locales/import
```

## 缓存策略

`ui_locale` 不参与以下 hash：

- `compute_translation_config_hash`
- `compute_ftbquests_config_hash`
- 资源包输出 hash
- FTB Quests source hash

原因：界面语言只影响展示，不影响翻译内容。

需要注意报告生成：

- 如果命中翻译缓存，仍应按当前 `ui_locale` 重新生成报告。
- 不应因为复用旧缓存而复用旧语言报告。
- 如果未来缓存报告 HTML，需要把 `ui_locale` 加入报告缓存 key，而不是翻译缓存 key。

直接 JSON 翻译模式可以复用现有翻译缓存，但缓存 key 必须包含：

- 输入 JSON 的 source hash。
- `source_locale`。
- `target_locale`。
- provider/model/API 参数 hash。
- glossary hash。
- JSON schema 类型：`flat` 或 `ui_locale`。

不应包含：

- 当前界面 `ui_locale`。
- 语言包导入时间。
- 浏览器主题。

## 实施步骤

### P0：前端体系语言切换

1. 新增 `uiLocales`、`uiMessages`、`t()`、`formatMessage()`、`applyUiLocale()`。
2. 在 header 增加体系语言下拉控件。
3. 新增隐藏字段 `ui_locale`。
4. 增加 localStorage 持久化和浏览器语言初始化。
5. 把首屏核心静态文案改为 `data-i18n`：
   - 标题和说明。
   - 导航。
   - 输入类型。
   - 语言设置。
   - 翻译器设置。
   - 开始按钮和取消按钮。
6. 把高频动态文案改为 `t()`：
   - 等待上传。
   - 上传中。
   - 处理完成。
   - 处理失败。
   - 文件选择提示。
   - 空状态。
7. 增加 Web UI 契约测试，确认：
   - 存在 `id="ui_locale"`。
   - 存在 `mc-mod-i18n-ui-locale`。
   - 存在体系语言切换控件。
   - `source_locale` 和 `target_locale` 仍保留原默认值。

### P1：服务端错误和报告本地化

1. 新增 `ui_i18n.py` 和语言 JSON 文件。
2. API 统一读取 `ui_locale`。
3. 将服务端用户可见错误改为错误 key。
4. 进度接口尽量返回状态 code，前端负责翻译展示。
5. 报告生成函数接收 `ui_locale`。
6. HTML 报告主要标题、表头、状态标签本地化。
7. JSON 报告保留字段名，增加 label 字段。
8. 增加测试：
   - `translate_ui()` fallback。
   - `ui_locale=en_us` 时错误消息为英文。
   - `ui_locale=en_us` 时 HTML 报告出现英文表头。

### P2：完整文案治理

1. 将 `INDEX_HTML` 中剩余硬编码文案逐步迁移到 key。
2. 增加脚本检查未登记文案或缺失翻译 key。
3. 支持 `zh_tw`。
4. 增加语言包贡献说明。
5. 如果单文件 HTML 维护成本过高，再考虑把语言包拆成静态 JSON endpoint。

### P3：扩展语言包与直接 JSON 翻译

1. 新增 `ui_locale` 导入导出 API。
2. 实现 `en_us.json` 基准导出。
3. 实现扩展语言包导入、校验和缓存落盘。
4. 增加“语言 JSON 文件”输入类型。
5. 实现扁平 JSON / UI schema JSON 的解析与翻译。
6. 实现 key 完整性、占位符、转义字符和 JSON 结构校验。
7. 增加“只补缺失 key”“覆盖全部 value”两种输出模式。
8. 把扩展语言包接入 UI 语言下拉框和 fallback 逻辑。

## 测试计划

### 单元测试

- `normalize_ui_locale("zh-CN") == "zh_cn"`。
- 未知语言回退到 `zh_cn`。
- 缺失 key 回退到默认语言。
- 占位符格式化正确。

### Web UI 契约测试

- 页面包含体系语言切换控件。
- 页面包含 `ui_locale` 隐藏字段。
- 页面包含 localStorage key。
- 页面仍包含：
  - `source_locale` 默认 `en_us`。
  - `target_locale` 默认 `zh_cn`。
  - FTB Quests 上传自动识别源语言函数。

### API 测试

- 不传 `ui_locale` 时默认中文。
- 传 `ui_locale=en_us` 时用户可见错误返回英文。
- 传非法 `ui_locale` 时回退中文。
- `ui_locale` 不改变翻译 config hash。
- 导出英文基准语言包返回合法 JSON。
- 导入扩展语言包后可在 `GET /api/ui-locales` 中看到。
- 直接 JSON 翻译模式能保留所有 key。
- 直接 JSON 翻译模式能保留 schema 元数据。

### 手工验收

1. 打开页面，默认中文。
2. 切换到 English，首屏标题、按钮、表单标签、状态提示立即变英文。
3. 刷新页面后仍为 English。
4. 切回中文后刷新仍为中文。
5. 切 English 后上传 `zh_cn.snbt`，源语言自动选择 `zh_cn`，界面仍保持 English。
6. 切 English 后目标语言仍默认 `zh_cn`。
7. 执行一次 JAR 翻译，结果摘要和主要按钮为 English。
8. 切中文后再次生成报告，报告标题和表头为中文。
9. 下载英文基准语言包，用它翻译出一个 `fr_fr.json` 并成功导入。
10. 上传 `en_us.json`，选择 `fr_fr`，输出的目标 JSON 保留全部 key。
11. 上传 UI 语言包 JSON 后，界面语言下拉框能出现新语言。

## 风险与处理

### 风险 1：硬编码文案很多，迁移容易漏

处理：

- P0 只覆盖高频路径。
- P1/P2 分批替换。
- 增加静态测试检查关键控件。

### 风险 2：体系语言和内容语言混淆

处理：

- 字段命名强制使用 `ui_locale`。
- UI 文案中避免只写“语言”，应写“界面语言”或“源语言/目标语言”。
- 测试覆盖切换 UI 语言不改变源语言/目标语言。

### 风险 3：缓存复用导致报告语言不一致

处理：

- 翻译缓存不包含 `ui_locale`。
- 报告每次按当前 `ui_locale` 重新生成。
- 如果未来缓存报告，单独使用 `report_locale` 维度。

### 风险 4：英文文案长度影响布局

处理：

- 按英文最长文本检查按钮、下拉框、表格表头。
- 保持控件有合理 `min-width` 和换行策略。
- 移动端检查无重叠。

### 风险 5：错误信息 key 化后排查变难

处理：

- API 返回同时包含 `error_code` 和本地化 `error`。
- 日志记录原始异常和错误 key。
- 前端展示本地化消息，开发排查看 code。

### 风险 6：语言包 schema 和普通 lang JSON 混用

处理：

- 通过 `schema_version` 和 `messages` 字段区分 UI 语言包。
- 普通 Minecraft lang JSON 仍按扁平对象处理。
- 导入时自动识别并在结果里明确标注类型。

### 风险 7：小语种语言包缺 key 影响体验

处理：

- 默认 fallback 到 `en_us` 或 `zh_cn`。
- 导入时输出缺失 key 清单。
- 界面显式显示“当前语言不完整”状态。

## 验收标准

1. 用户能在页面顶部切换界面语言。
2. 语言选择会持久化到本地，下次打开仍生效。
3. 切换界面语言不会改变源语言、目标语言和翻译结果。
4. `source_locale` 仍默认 `en_us`，`target_locale` 仍默认 `zh_cn`。
5. 上传 `zh_cn.snbt` 时，源语言仍能自动识别为 `zh_cn`。
6. 主要操作链路的文案随体系语言切换。
7. 传入 `ui_locale=en_us` 的任务，其主要错误信息和报告标题可显示英文。
8. 原有 Web UI 契约测试通过。
9. 扩展语言包可导入、可选、可 fallback。
10. 直接上传 JSON 可翻译为目标语言并保留 key。

## 建议优先级

建议先做 P0。原因是当前项目的 Web UI 仍是单文件架构，直接一次性全量国际化成本高、容易引入回归。P0 可以先建立 `ui_locale` 的边界、入口、持久化和核心文案切换，为后续服务端报告和错误本地化打基础。

P0 完成后再做 P1，把错误信息和报告纳入体系语言。随后做 P2 清理剩余硬编码文案。P3 再落地扩展语言包和直接 JSON 翻译，这样可以先把“界面可切换”做稳，再把“语言文件可导入导出”做完整。
