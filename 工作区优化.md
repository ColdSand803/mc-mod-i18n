# MC-LocaliZ 工作区优化方案

## 1. 结论

工作区应该做减法，但减的不是功能，而是同屏竞争、重复表达和过早暴露的复杂度。

当前工作区已经具备输入、语言、翻译器、预检、输出策略、任务进度、结果审校、失败重试等关键能力。问题已经不再是“缺 UI”，而是用户第一次进入时仍然容易把它理解成一张长配置表，而不是一个清晰的本地翻译工作台。

本方案的核心方向是：

- 保留“两栏工作台”：左侧配置，右侧结果
- 把左侧配置改成线性任务流程
- 把高级项和长期设置继续压到第二层
- 让预检成为启动前的自然步骤
- 让结果区优先回答“能否下载、是否失败、是否需要复核”

一句话概括：做流程减法，不做功能减法。

---

## 2. 优化目标

### 2.1 用户目标

用户进入工作区后，应该能很快完成这条路径：

```text
选择输入 -> 选择语言和翻译器 -> 调整可选项 -> 运行预检 -> 开始生成 -> 查看结果 -> 下载或复核
```

首屏不追求“展示系统有多强”，而是要让用户知道：

- 我现在该先选什么
- 哪些选项是必填
- 哪些选项可以先不管
- 开始前有没有风险
- 完成后下一步该下载还是修复

### 2.2 产品目标

工作区应该像一个稳定的本地工具，而不是管理后台或数据大屏。

它需要：

- 安静，不抢注意力
- 直接，主操作始终明确
- 可解释，不同 provider 的成本和风险要提前出现
- 可复核，结果页优先暴露失败项和风险项
- 可维护，继续适配当前 `web.py` 单文件架构，不引入前端框架重写

---

## 3. 设计原则

### 3.1 少即是清楚，不是少功能

不要为了显得专业而增加顶部搜索、通知、头像、统计卡片、大图背景、多层导航。它们会让主流程变钝。

工作区的专业感应该来自：

- 清晰的步骤
- 稳定的控件
- 明确的状态
- 可靠的错误反馈
- 可预测的结果入口

### 3.2 配置区流程化

左侧配置区不再被理解为一堆平级表单，而应被理解为四个步骤：

1. 选择输入
2. 选择语言与翻译器
3. 调整可选增强项
4. 运行预检并开始

不建议做真正的多页 wizard。当前工具面向本地处理和反复调试，多页切换会降低效率。更合适的是“单页流程”，用步骤标题、分组和状态来建立顺序感。

### 3.3 结果区渐进展开

右侧结果区在任务未开始时应该保持轻，不需要预置大量指标卡。

任务完成后再展开：

- 下载入口
- 失败和风险摘要
- 结果表格
- 对比审校
- 报告
- API 日志
- 硬编码映射

结果区的第一优先级不是“展示所有数据”，而是帮用户做下一步决策。

### 3.4 长期配置不要回流到工作区

工作区只放“本次任务需要决定”的东西。

设置页继续承接：

- 缓存维护
- 配置预设
- 翻译记忆
- 全局术语表
- UI 语言包
- 长期 provider 默认值

不要把设置页已经承担的长期管理能力重新塞回工作区。

---

## 4. 当前布局判断

### 4.1 保留的方向

当前真实工作区已经有几个正确方向：

- 左侧配置、右侧结果的主结构是合理的
- 高级 API 设置已经折叠，是正确的
- 预检已经成为独立卡片，是很好的基础
- 任务历史、文档、设置已经从主流程中分离
- 结果页已有表格、对比、报告、日志、硬编码等能力

这些不需要推翻。

### 4.2 需要避免的方向

不建议把工作区做成排版 mock 里那种强 dashboard 形态：

- 固定大侧栏
- 顶部二级导航
- 全局搜索
- 通知和用户头像
- 多个 KPI 卡片
- 大面积视觉图
- 配置、结果、指标同时争抢首屏

这类布局看起来“完整”，但对 MC-LocaliZ 的核心任务来说过重。用户不是来巡检系统运营数据，而是来完成一次翻译任务。

---

## 5. 建议的信息架构

### 5.1 全局框架

保留当前应用框架：

- 左侧主导航：工作区、任务历史、翻译报告、硬编码、API 日志、设置、文档、帮助
- 顶部栏：品牌、当前任务、界面语言、主题、少量状态
- 主工作区：配置面板 + 结果面板

顶部栏不建议新增搜索、通知、用户头像等通用 SaaS 元素。

### 5.2 工作区主结构

建议工作区保持两栏：

```text
+--------------------------+--------------------------------+
| 配置面板                 | 结果面板                       |
|                          |                                |
| Step 1 输入              | 未开始：轻量空状态             |
| Step 2 语言与翻译器      | 进行中：当前任务进度           |
| Step 3 可选增强项        | 完成后：下载 / 风险 / 审校     |
| Step 4 预检并开始        |                                |
+--------------------------+--------------------------------+
```

配置面板负责“发起任务”，结果面板负责“观察和处理任务结果”。两边职责不要混杂。

---

## 6. 配置面板优化

### 6.1 Step 1：选择输入

内容：

- 输入类型：Mod JAR、FTB Quests、语言 JSON
- 对应文件选择控件
- 已选文件摘要

减法建议：

- 输入类型切换保持在第一个卡片，不和 provider、输出策略混在一起
- 文件选择后显示简洁摘要，不急着展示完整文件列表
- 模式切换后只显示当前模式需要的文件控件

目标：

用户先确认“我要处理什么”。

### 6.2 Step 2：选择语言与翻译器

内容：

- 源语言
- 目标语言
- 翻译器 provider
- 资源包格式
- provider 风险提示

减法建议：

- 把语言和 provider 归为同一步，因为它们共同决定翻译策略
- provider 下面直接显示提示条，不藏到高级设置里
- 离线 provider 的 UI 要显得轻，AI provider 的风险和成本要更明确

provider 提示文案示例：

- `复制原文`：不翻译，仅用于验证流程和打包输出
- `离线术语表`：稳定、无费用，但覆盖有限
- `Argos Translate`：本地离线翻译，首次配置可能需要安装语言包
- `Deep Translator`：免费接口可用性不稳定，适合轻量尝试
- `OpenAI 兼容`：质量较高，需要 API Key，可能产生费用
- `Anthropic 兼容`：质量较高，需要 API Key，可能产生费用

目标：

用户在填 API Key 之前就知道自己选择的复杂度。

### 6.3 Step 3：可选增强项

内容：

- 本次任务术语表上传
- 输出策略
- 高级 API 设置

减法建议：

- 术语表是增强项，不要和输入文件形成同等视觉重量
- 输出策略默认折叠，但保留当前选择摘要
- 高级 API 设置只在 AI provider 下显著提示
- 对 `copy`、`glossary` 等轻量 provider，弱化 API 区存在感

输出策略摘要示例：

```text
输出策略：保留已有中文、使用缓存、扫描硬编码
```

目标：

用户知道这些是“可以改”，但不是“必须先懂”。

### 6.4 Step 4：预检并开始

内容：

- 运行预检按钮
- 预检摘要
- warning / blocker 列表
- 开始生成按钮
- 当前状态一句话

减法建议：

- 预检未运行时，显示“建议先运行预检”
- 预检完成后，固定展示摘要 chips
- 开始按钮文案随预检状态变化
- 状态框只显示一句主状态，不重复 loading card 的详细信息

开始按钮状态示例：

- 未预检：`开始生成`
- 预检中：`正在预检...`
- 有阻断：`存在阻断，需先处理`
- 无阻断：`开始生成（已通过预检）`
- 运行中：`正在生成...`

目标：

预检不再像附属工具，而是启动前的自然步骤。

## 7. 结果面板优化

### 7.1 未开始状态

未开始时不要展示复杂 dashboard。

建议只保留：

- 一句空状态说明
- 最近一次任务入口，如果已有历史任务
- 可选的“选择输入后将在这里显示结果”

目标：

让用户注意力留在左侧配置流程。

### 7.2 进行中状态

进行中状态只聚焦当前任务：

- 当前阶段
- 当前文件
- 文件进度
- 请求进度
- 缓存命中
- 重试原因
- 取消任务

顶部 header、状态框、loading card 要明确分工：

- header：当前任务 ID / 全局状态
- 配置区状态框：一句摘要
- loading card：详细进度

不要三个地方都讲同一件事。

### 7.3 完成状态

完成后结果区第一屏按优先级展示：

1. 是否可下载
2. 是否有失败项
3. 是否有风险项或需人工复核项
4. 主要产物入口
5. 详细表格和对比视图

推荐结构：

```text
结果摘要
- 下载资源包
- 查看报告
- 失败 0 / 风险 3 / 已翻译 1280

优先处理
- 有失败项：只看失败项 / 重试失败项
- 有风险项：只看风险项 / 打开对比审校

审校区
- 表格视图
- 对比视图
- API 日志
- 硬编码映射
```

目标：

用户先知道“能不能交付”，再进入细节。

### 7.4 默认视图规则

建议结果页默认视图按任务质量自动选择：

- 有失败项：默认显示失败 / 风险优先处理区
- 有风险项但无失败项：默认显示对比视图
- 全部正常：默认显示表格视图和下载入口

这能减少用户在多个结果入口之间犹豫。

---

## 8. 视觉与交互约束

### 8.1 保持工具感

视觉关键词：

- 专业
- 稳定
- 紧凑
- 清晰
- 低噪音

不建议：

- 大面积装饰图
- 过多渐变
- 过多统计卡
- 营销式 hero
- 一屏多个主按钮

### 8.2 控件密度

工作区可以保持较高信息密度，但要有明确分组。

建议：

- 卡片标题短，说明文字少
- 必填项优先排列
- 可选项折叠
- 操作按钮贴近它影响的区域
- 长说明跳转到文档，不在工作区展开

### 8.3 可访问性

至少保证：

- 所有交互控件有清晰 focus 状态
- 文本对比度达到常规可读标准
- 图标按钮有可理解标签或 `aria-label`
- 移动端不出现横向滚动
- 预检 warning 和 blocker 不只依赖颜色区分

---

## 9. 不建议做的事

### 9.1 不做完整多页 wizard

原因：

- 本地工具需要反复调整参数
- 用户可能需要同时看配置和结果
- 多页切换会增加回看成本

可以做单页步骤，不做切屏流程。

### 9.2 不做强 SaaS 顶栏

不建议加入：

- 工作区搜索
- 通知铃铛
- 用户头像
- 多级产品导航

这些不是当前任务核心，会让工具显得更重。

### 9.3 不把结果页继续堆成 KPI 仪表盘

结果页需要指标，但指标不是第一主角。

用户最关心的是：

- 下载什么
- 哪里失败
- 哪些要复核
- 如何重试

### 9.4 不引入前端框架重写

当前阶段的问题主要是信息结构和状态呈现，不是框架能力不足。

继续在当前 `web.py` 架构中做小步优化更稳。

## 10. 分阶段落地

### 第一阶段：流程减法

目标：让工作区从“配置表”变成“任务流程”。

事项：

- 调整步骤分组为 4 步
- 合并语言与 provider 的视觉关系
- 把术语表、输出策略、高级 API 归入可选增强项
- 简化顶部和首屏无关信息

验收标准：

- 首屏能看出下一步是先选输入
- 用户不用展开高级项也能启动常规任务
- `copy` / `glossary` 等轻量 provider 不显得像必须配置 API

### 第二阶段：预检前置

目标：让预检成为开始生成前的自然步骤。

事项：

- 预检摘要默认可见
- warning / blocker 数量前置
- 开始按钮根据预检状态变化
- 阻断项提供明确修改方向

验收标准：

- 开始前能一眼看到文件数、待翻译条目数、warning 和 blocker
- 有阻断时用户知道为什么不能继续
- 无阻断时主按钮明确提示可以生成

### 第三阶段：结果优先级

目标：让用户完成后更快进入下载、排错或复核。

事项：

- 增加结果摘要区
- 增加失败 / 风险优先处理区
- 有风险任务默认进入对比审校
- 统一失败项重试入口

验收标准：

- 完成后第一屏能看到下载入口
- 有失败项时失败入口比普通表格更显眼
- 有风险项时用户能直接进入复核视图

### 第四阶段：状态收束

目标：避免任务状态在多个位置重复表达。

事项：

- header 只保留任务 ID 和全局状态
- 配置区状态框只保留一句摘要
- loading card 承担详细进度
- 结果区承担完成后决策

验收标准：

- 同一个状态不会在三处用不同文案重复出现
- 用户知道看哪里获取详细进度
- 任务完成后状态自动让位给结果摘要

---

## 11. 最小版本建议

如果只做一轮优化，优先做这 5 件事：

1. 把左侧配置明确整理成 4 个步骤
2. provider 下方增加即时风险和成本提示
3. 预检摘要默认可见，并影响开始按钮状态
4. 结果完成后增加“下载 / 失败 / 风险”摘要
5. 有失败或风险时，把对应处理入口放到结果区最上方

这组改动不会推翻现有结构，但能显著降低首次使用成本。

---

## 12. 成功标准

优化完成后，工作区应该满足：

- 新用户不用读文档，也能理解主流程
- 常规任务在不展开高级项的情况下可以完成
- provider 的风险和成本不会等到报错后才暴露
- 预检结果能影响用户是否开始生成
- 结果页第一屏能判断是否可下载、是否失败、是否要复核
- 设置页和工作区边界清晰，没有长期配置回流

---

## 13. 最终判断

MC-LocaliZ 的工作区不需要变成更大的界面，而需要变成更明确的界面。

最值得做的是减少首屏噪音、强化流程顺序、收束任务状态、前置风险提示、突出结果决策。这样既能保留当前功能完整度，也能让产品更像一个真正可长期使用的本地翻译工作台。
