# 帮助和文档实施方案

## 1. 目标

把当前项目里分散的说明信息收口成一套真正可用的“帮助 / 文档”系统，让用户在 Web UI 内就能：

- 快速知道“这是什么功能，什么时候用”
- 遇到报错或预检阻断时能立即找到解释和处理步骤
- 不离开应用就能完成基础上手、provider 选择、参数理解和常见问题排查

这次的目标不是做一个泛化 CMS，而是先把当前产品里已经存在的帮助信息正式产品化。

---

## 2. 当前现状

从现有代码看，项目已经具备一部分“帮助素材”，但还没有形成完整功能：

### 2.1 已有基础

- 侧边栏底部已经有：
  - `nav.docs`
  - `nav.help`
- UI 中已经有大量上下文帮助文案：
  - `provider.*.help`
  - `advanced.*_help`
  - `preflight.*`
  - `settings.*`
  - `output.*`
- README 已经覆盖：
  - CLI 启动方式
  - provider 说明
  - Azure / LibreTranslate / Argos / deep-free 使用方式
  - 硬编码扫描与输出说明

### 2.2 当前缺口

- `文档` / `帮助` 入口只是占位，不可真正切换
- 帮助内容分散在：
  - `README.md`
  - `ui_i18n.py`
  - `web.py` 里的 `field-help`
  - 预检和状态提示
- 缺少统一信息架构：
  - 新手入口
  - 常见问题
  - provider 对比
  - 错误排查
  - 输出说明
- 缺少上下文联动：
  - 遇到预检阻断时没有“查看为什么”
  - 遇到 provider 失败时没有“查看处理方法”
  - 输出策略、缓存、翻译记忆等设置没有深度说明入口

## 3. 设计原则

### 3.1 先做“实用帮助”，不做“知识库平台”

第一期只做项目当前最需要的内容：

- 上手文档
- provider 说明
- 输出策略说明
- 预检说明
- 常见错误排查
- 报告 / 历史 / API 日志怎么看

不做：

- 富文本编辑后台
- 在线协作编辑
- 权限系统
- 全文搜索引擎

### 3.2 文档必须离线可用

这是本地工具，帮助系统必须在本地 UI 内直接可用，不能依赖外部网站。

### 3.3 帮助分两层

- `文档`
  - 面向“系统化阅读”
  - 适合新手入门、完整理解功能
- `帮助`
  - 面向“当前任务排障”
  - 适合用户正在某个页面、某个字段、某个错误上需要立即解释

### 3.4 帮助内容必须和 UI 状态联动

帮助不是单独页面堆文章，而是要接进：

- provider 选择
- 预检结果
- 输出策略
- 历史任务
- 报告页
- API 日志页

## 4. 功能范围

建议把“帮助和文档”拆成四块。

### 4.1 文档中心

侧边栏 `文档` 打开独立视图，展示结构化文档目录。

建议文档分类：

- 快速开始
- 翻译器选择
- 输出策略
- 预检与阻断
- 翻译记忆 / 缓存
- 报告、历史与 API 日志
- 硬编码扫描
- 常见问题

### 4.2 帮助中心

侧边栏 `帮助` 打开“问题导向”视图。

建议内容形式：

- 常见问题卡片
- 按场景分类
  - 为什么预检失败
  - 为什么没生成中文
  - 为什么 provider 测试连接失败
  - 为什么命中缓存 / 命中记忆后结果没变化
  - 为什么历史任务能看到但文件下载不到
- 快捷跳转到相关页面或文档章节

### 4.3 上下文帮助

在现有页面中增加“查看详细说明”能力，而不是只保留一句 tooltip。

重点区域：

- provider 说明区
- BaseURL / Region / API Key 环境变量
- 并发、重试、批大小、超时
- 输出策略
- 预检摘要
- 翻译记忆设置
- 历史任务状态

### 4.4 错误帮助联动

当系统出现状态或错误时，直接提供“为什么”和“怎么处理”的入口。

例如：

- 预检阻断
- provider smoke test 失败
- API 401 / 429 / timeout
- Argos 缺依赖 / 缺语言包
- Azure region 配置错误
- 下载文件不存在

## 5. 信息架构建议

### 5.1 文档中心目录

建议文档中心左侧目录或顶部目录包含：

1. 快速开始
2. 输入类型说明
3. 翻译器选择指南
4. 高级 API 设置
5. 输出策略
6. 预检机制
7. 翻译记忆与缓存
8. 结果页 / 翻译报告
9. 任务历史
10. API 日志
11. 硬编码工作台
12. FAQ

### 5.2 帮助中心入口分类

建议帮助中心用“问题分类”而不是纯文章目录：

- 我第一次使用，不知道从哪开始
- 我不知道选哪个翻译器
- 我已经开始翻译，但结果不对
- 我遇到了连接 / API 错误
- 我想理解缓存、记忆、重试等高级选项
- 我想导出、排查和复用历史任务

### 5.3 文档主题标识

每篇文档统一元信息：

- `id`
- `title`
- `category`
- `summary`
- `keywords`
- `related_topics`
- `applies_to`
  - `workspace`
  - `history`
  - `report`
  - `api-log`
  - `provider:<name>`
  - `feature:<name>`

## 6. 技术实现建议

## 6.1 内容存储

建议使用“本地 Markdown + 索引清单”的方式。

建议新增目录：

- `docs/help/`
  - `index.json`
  - `quick-start.md`
  - `providers.md`
  - `preflight.md`
  - `output-strategy.md`
  - `translation-memory.md`
  - `history-and-report.md`
  - `hardcoded.md`
  - `faq.md`

原因：

- 易维护
- 易做版本管理
- 与当前仓库文档习惯一致
- 后续可继续导出到 README 或网页

### 6.2 服务端接口

建议在 `web.py` 增加文档读取接口：

- `GET /api/docs`
  - 返回文档目录、标题、摘要、分类
- `GET /api/docs/<slug>`
  - 返回单篇 Markdown 内容和元信息
- `GET /api/help/context?topic=...`
  - 返回某个上下文帮助块

第一期也可以更简单：

- 把文档内容直接在服务端启动时读入内存
- 前端只请求目录和正文

### 6.3 前端视图

建议新增两个独立视图：

- `docs-page`
- `help-page`

接入 `switchView(view)`，与现有：

- `language`
- `history`
- `report`
- `hardcoded`
- `api-log`
- `settings`

保持一致。

### 6.4 Markdown 渲染

建议第一期只支持有限 Markdown：

- 标题
- 段落
- 列表
- 代码块
- 表格
- 链接

可以选两种路线：

#### 路线 A：服务端预渲染 HTML

优点：

- 前端逻辑简单
- 便于统一样式

缺点：

- 服务端要额外处理 Markdown

#### 路线 B：前端渲染 Markdown

优点：

- 文档接口更干净
- 更容易做目录和搜索

缺点：

- 前端多一层渲染逻辑

建议第一期选 `路线 A`。

原因：

- 当前项目核心 UI 都在 `web.py`
- 第一阶段重点是尽快可用
- 服务端渲染更容易控制安全性和样式一致性

### 6.5 搜索

第一期不需要全文搜索引擎。

建议先做轻量搜索：

- 按标题 / 关键词 / 摘要过滤
- 前端内存筛选

等文档量上来以后再考虑全文搜索。

## 7. 内容设计建议

### 7.1 第一批必须完成的文档

#### 1. 快速开始

解决：

- 第一次打开 UI 后怎么操作
- 先选什么输入
- 什么时候先跑预检
- 如何开始一次最小翻译任务

#### 2. 翻译器选择指南

解决：

- `copy` / `glossary` / `deep-free` / `libretranslate` / `argos` / `azure-translator` / OpenAI / Anthropic 分别适合什么
- 哪些需要联网
- 哪些需要 API Key
- 哪些适合零配置试用

#### 3. 高级 API 设置

解决：

- BaseURL 怎么填
- Region 什么时候需要
- API Key 环境变量与明文填写的区别
- 并发 / 重试 / 批大小 / 超时应该怎么调

#### 4. 输出策略

解决：

- 覆盖已有中文
- 跳过已翻译 JAR
- 忽略缓存
- 忽略翻译记忆
- 忽略预检阻断
- 扫描硬编码

#### 5. 预检机制

解决：

- 为什么建议先跑预检
- “阻断”和“警告”的区别
- 常见阻断项有哪些

#### 6. 翻译记忆与缓存

解决：

- checkpoint 缓存和 translation memory 的区别
- 为什么“结果没变化”
- 什么时候该清缓存
- 什么时候该忽略翻译记忆

#### 7. 结果 / 报告 / 历史 / API 日志

解决：

- 报告页怎么看
- 历史任务能做什么
- API 日志什么时候打开
- 下载文件不存在是什么意思

#### 8. FAQ

解决高频问题：

- 为什么测试连接失败
- 为什么没生成资源包
- 为什么有些词没翻
- 为什么生成结果里还有英文
- 为什么历史文件打不开

### 7.2 上下文帮助模板

建议统一结构：

- 这个字段 / 功能是干什么的
- 什么时候需要改
- 常见误区
- 推荐值 / 推荐做法
- 查看完整文档

例如 `并发`：

- 用途：控制并发批次数
- 何时修改：大量文本、网络较快时
- 风险：中转站限流、429、503
- 推荐：默认按提示，异常时降到 1

## 8. UI 交互方案

### 8.1 侧边栏入口

把当前占位的：

- `文档`
- `帮助`

改成真正可点击 `button.nav-item`，并加入 `switchView()`。

### 8.2 文档页布局

建议：

- 左侧：目录 / 分类
- 右侧：正文
- 顶部：搜索框 + 面包屑

如果不想加双栏复杂布局，第一期也可以：

- 上方分类 tabs
- 下方目录列表
- 点击后显示正文

### 8.3 帮助页布局

建议做成“场景卡片 + 热门问题”：

- 我是第一次使用
- 我需要选翻译器
- 我遇到连接错误
- 我看不懂结果和历史
- 我想理解缓存 / 记忆 / 输出策略

点击卡片后展开对应帮助主题。

### 8.4 页面内跳转帮助

给关键区域加：

- `?`
- `查看说明`
- `了解更多`

跳转到：

- 帮助页的具体主题
- 或文档页具体文章锚点

重点先接这些位置：

- provider 选择
- API 高级设置
- 输出策略
- 预检结果
- 翻译记忆设置
- 历史任务状态

### 8.5 错误提示联动

对明确错误增加帮助链接，例如：

- `连接测试失败：401 Unauthorized`
  - 附 `查看认证错误说明`
- `缺少 Argos 语言包`
  - 附 `查看离线模型准备步骤`
- `预检发现阻断项`
  - 附 `查看预检阻断说明`

## 9. 推荐实施路线

建议分四期做。

### 第一期：把入口做活

目标：

- `文档` / `帮助` 侧边栏入口可点击
- 增加独立页面
- 先放静态内容

本期产出：

- docs/help 内容目录
- 文档 / 帮助页面骨架
- 基础 API
- 基础 Markdown 渲染

### 第二期：把已有说明收口

目标：

- 把 README、provider help、field-help、预检说明整理成正式文档

本期产出：

- 快速开始
- provider 说明
- 输出策略说明
- 预检说明
- 翻译记忆 / 缓存说明
- FAQ 初版

### 第三期：做上下文联动

目标：

- 让用户在当前页面就能跳到对应说明

本期产出：

- provider “查看说明”
- 输出策略“查看说明”
- 预检摘要“为什么会阻断”
- 历史任务 / API 日志的上下文帮助

### 第四期：做错误导向帮助

目标：

- 让错误信息可操作，而不是只显示原始报错

本期产出：

- 错误码到帮助主题映射
- provider 常见失败解释
- 推荐修复步骤

## 12. 风险与取舍

### 12.1 最大风险

不是技术，而是内容维护。

如果文档系统做出来，但内容不更新，很快会比没有文档更糟。

### 12.2 对策

文档第一期必须只覆盖当前已稳定功能，不写未来态空话。

每次新增功能时要求同步补：

- 一条帮助文案
- 一段文档说明
- 一条 FAQ 或排障说明（如有必要）

### 12.3 不建议的做法

- 直接把整个 README 原样塞进 UI
- 只做帮助页面，不做上下文联动
- 只做 tooltip，不做完整文档
- 一开始就上复杂全文搜索

---

## 13. 推荐结论

建议按下面路线推进：

1. 先把 `文档` / `帮助` 入口做成真正页面
2. 用 `docs/help/*.md + index.json` 做本地离线文档源
3. 第一批内容先收口当前已有说明，不扩张范围
4. 第二阶段再把帮助与 provider、预检、输出策略、历史任务联动起来
5. 最后做错误导向帮助

一句话总结：

> 这项功能最合理的做法，不是“再加几个说明文字”，而是把现有 README、tooltip、provider help、预检提示、设置说明，统一收口成一个本地离线、可跳转、可上下文联动的帮助与文档系统。
